JDK25 与 Leyden 相关进展记录

Posted on 2024-10-05 Edited on 2025-12-12 In Java Views:

这篇是 JDK21 升级之后继续关注的一些方向，主要包括 Loom、JVM、JFR、GC、对象头压缩，以及 Leyden。

JDK21 是一个 LTS 落点，但很多能力并不是到 JDK21 就结束了。实际做升级时，还是要看后续几个版本在解决什么问题。

Loom（虚拟线程）

Release JEP Description
24 JEP 491: Synchronize Virtual Threads without Pinning 解决了虚拟线程在 synchronized 上绑定的问题。虚拟线程从此可以更大规模使用了，但类加载等场景依然需要关注。
25 JEP 506: Scoped Values 引入一种全新的、基于显式作用域管理的值传递方式。

JDK21 里虚拟线程已经正式可用，但如果从生产大规模使用角度看，我认为 JDK24 的 JEP 491 非常关键。

JVM 相关

Release JEP Description
25 JEP 519: Compact Object Headers 完成 Lilliput 项目的第一阶段，将对象头从 128 位减小到 64 位，下一阶段目标是 32 位。对象很多的应用可能会有内存收益，具体效果和项目本身关系比较大。
25 JEP 521: Generational Shenandoah Shenandoah 也进入分代方向。不过 JDK25 里默认仍是单代 Shenandoah。低延迟 GC 后面基本都在利用分代假说继续优化。
25 JEP 520: JFR Method Timing & Tracing 支持对指定方法做更精确的耗时统计和 tracing。这个对线上分析很有吸引力，但生产开启范围和开销要认真控制。
23 / 24 JEP 474: ZGC: Generational Mode by Default / JEP 490: ZGC: Remove the Non-Generational Mode ZGC 后续只有分代模式了。实测我认为绝大部分情况下分代 ZGC 优于单代 ZGC，少部分不符合分代假说的极端情况除外。

语法 & core-lib

Release JEP Description
24 JEP 485: Stream Gatherers 提供更灵活的 stream 流处理方式，并且提供了一些内置扩展。
22 JEP 456: Unnamed Variables & Patterns 可以使用下划线 _ 来命名未使用的变量。
23 JEP 467: Markdown Documentation Comments 注释支持 Markdown 格式。

底层库

Release JEP Description
22 JEP 454: Foreign Function & Memory API 提供标准 FFM API，用于调用其他语言代码以及本地内存管理。
24 JEP 484: Class-File API 提供用于解析、生成和转换 Java 类文件的标准 API。

Leyden 相关

DONE

  • JEP 483: Ahead-of-Time Class Loading & Linking
  • JEP 514: Ahead-of-Time Command-Line Ergonomics
  • JEP 515: Ahead-of-Time Method Profiling
  • JEP 516: Ahead-of-Time Object Caching with Any GC

这里和我最初整理时有一个变化：JEP 516 已经是 JDK26 delivered，不再是 doing 状态。所以如果只讨论 JDK25，需要把它放到“后续继续关注”的位置；如果讨论 Leyden 路线，就可以一起看。

DOING / 继续关注

  • AOT Code Compilation 相关草案。
  • 对更多 class loader 场景的支持。
  • 和 Spring Boot 打包、jar 展开、CDS 的工程化结合。
  • 启动参数、训练流程和正式启动流程的隔离。

核心 JEP 解读

JEP 483

JEP 483: Ahead-of-Time Class Loading & Linking

存储类加载和链接的训练数据。它关注的是应用启动时反复做的类加载、解析、链接工作，能不能提前做一部分。

tips：

对 lambda 等的优化会比较大。

需要注意的是，它不是把 Java 变成 native 可执行文件，也不是直接消除 JVM 动态性。它更像是让 JVM 在启动时少做一部分重复工作。

JEP 515

JEP 515: Ahead-of-Time Method Profiling

存储方法调用数据，例如方法执行次数等，以便 JIT 了解方法执行情况，可以直接进行对应层级的编译优化。

tips：

这样做需要注意，由于 JIT 已知方法执行情况，理想情况下可以认为热点 path 代码可以在首次触达时即完成 JIT 优化。这对 CPU 资源的开销可能会比较大。

也就是说，如果直接接入峰值流量，启动初期消耗的 CPU 资源可能较高，可能依然需要慢启动流程。不过流量放大的速度可以比现在快很多。

未来 AOT Code Compilation 完成后，上述问题可能会继续改善。

JEP 514

JEP 514: Ahead-of-Time Command-Line Ergonomics

作用是改善 AOT 命令行体验，把训练和生成配置的流程包装得顺一点。

这个 JEP 本身不算最核心的性能能力，但对工程化很重要。AOT 相关能力如果使用成本太高，就很难真正进入普通服务发布流程。

目前需要注意的是：生成训练数据时的启动参数和正式启动应用时不应该简单完全复用。例如管理端口、监控端口、临时目录、文件路径，都可能在训练生成阶段产生冲突。

JEP 516

JEP 516: Ahead-of-Time Object Caching with Any GC

JEP 516 是 JDK26 的内容，不属于 JDK25，但它很能说明 Leyden 的方向：不只是类加载和链接，后续还会把一些对象状态提前缓存。

这个方向对启动和预热会有更大帮助。但具体到业务应用收益、和应用框架如何结合、训练数据如何设计，仍然需要继续观察。

目前最大的问题

class loader

Leyden 相关能力很容易受 class loader 影响。

简单 classpath 应用是一回事，真实服务经常还有：

  • fat jar / nested jar。
  • 应用服务器 class loader。
  • 插件化 class loader。
  • 框架自定义 class loader。
  • agent 或字节码增强。

所以不要只看简单 demo 的效果。真实服务要看自己的打包方式和类加载路径。

Spring Boot 这类框架已经在 jar 展开、CDS、AOT 方向做了一些工作，普通服务优先看框架集成会更稳。

相关资料：

  • https://spring.io/blog/2024/08/29/spring-boot-cds-support-and-project-leyden-anticipation
  • https://docs.spring.io/spring-boot/reference/packaging/efficient.html
  • https://docs.spring.io/spring-boot/docs/3.2.9/reference/html/executable-jar.html#appendix.executable-jar.nested-jars

GC 支持

早期 Leyden/AOT 能力和 GC 的配合有一些限制，后续 JEP 516 已经推进到 Any GC。

对业务应用来说，比较理想的状态是：不需要为了启动优化放弃原本选择的 GC。

训练流量

AOT profile 类能力依赖训练数据。训练数据是否代表真实流量，会直接影响收益。

如果训练流量和真实流量差异很大，可能只优化了启动路径，没优化真正的业务热点路径。对复杂服务来说，训练阶段如何设计是一个工程问题，不只是 JVM 参数问题。

Leyden 和 Native Image 的区别

Leyden 相比 Native Image 而言，不会砍掉 JVM 的动态性。对开发而言可以做到基本无感。

Leyden 项目不打算直接合并 GraalVM 的工作，而是参考一部分 Graal 思路的同时，在项目开始时保留一部分运行时特征。

相关背景：

  • https://www.infoq.com/news/2022/06/project-leyden-delays-aot/
  • https://mail.openjdk.org/pipermail/discuss/2020-April/005429.html
  • https://mail.openjdk.org/pipermail/leyden-dev/2022-May/000005.html
  • https://openjdk.org/projects/leyden/notes/01-beginnings

我的理解是：

  • Native Image 更激进，收益明显，约束也明显。
  • Leyden 更温和，更适合大量依赖 JVM 动态性的传统 Java 服务。

需要关注的工作

  1. JEP 491 后，虚拟线程在真实服务里的风险是否明显降低。
  2. JFR Method Timing & Tracing 的生产使用方式和开销。
  3. Compact Object Headers 对对象密集型服务的实际内存收益。
  4. ZGC/Shenandoah 分代化之后的真实吞吐和延迟表现。
  5. Leyden 和 Spring Boot、CDS、jar 展开的集成路径。
  6. AOT profile/缓存能力对启动、预热、扩容速度的改善。

JDK21 升级完以后，不代表这条线结束。JDK21 是一个适合落地的版本；JDK24/JDK25/JDK26 展示的是后续演进方向。
