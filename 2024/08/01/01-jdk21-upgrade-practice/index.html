JDK21 升级适用指南

Posted on 2024-08-01 Edited on 2025-06-15 In Java Views:

当前升级 JDK 的动力实际上主要来自三项：ZGC、虚拟线程、衍生生态。

以下对 JDK 高版本主要特性做一个总览分析。这里不写成“JDK21 新特性大全”，只记录我在实际升级时比较关心的收益、适用场景和风险。

1. 总体分析

ZGC

优点：ZGC 基本消除了 GC 带来的延迟，并且对大对象有更好的处理。

不足：ZGC 吞吐量比不上 G1。即便分代 ZGC 吞吐量有了很大提升，但与 G1 依然有一定差距。

我的理解是，如果应用的问题主要是 GC pause 带来的长尾，ZGC 值得优先验证。如果应用本身更看重极限吞吐，或者 GC 本来就不是瓶颈，那就不要只因为“新”而切。

虚拟线程

优点：适用 IO 密集型应用。一方面可以提高系统性能，另一方面可以极大消除线程带来的内存压力。

不足：在目前 JDK21 的版本中，虚拟线程存在的坑还比较多，个人不建议在可靠性要求较高的应用上直接大规模使用。

尤其需要注意 synchronized 与虚拟线程绑定平台线程的问题。轻则降低吞吐量，严重可能导致死锁。虚拟线程可以灰度，但不要和 JDK 升级本身混成一个变量。

衍生生态

许多 Java 生态框架都基于高版本 JDK 开发，并不再维护低 JDK 版本。如果想要探索基于高版本的框架，那么升级 JDK 也是势在必行的了。

这类收益不一定马上体现在接口耗时上，但会体现在后续框架升级、依赖治理、工具链维护成本上。

2. 什么应用适合升级 JDK？

目标 适用应用 具体版本 风险
ZGC GC 延迟敏感的应用，例如缓存应用；本地有大缓存、频繁生成大对象的应用 如果对吞吐量追求不高，可以考虑 JDK17 以上；如果对吞吐量有较高要求，需要升级 JDK21，并且开启分代特性 分代 ZGC 需要经过充分验证后才可上线
虚拟线程 IO 密集型应用都可以评估，文件 IO 场景需要单独验证 JDK21 synchronized 兼容不好时会导致虚拟线程与平台线程绑定，轻则降低吞吐量，严重可能导致死锁
衍生生态 需要按自己实际调研进行升级考虑，例如新框架、新工具链、新运行时能力 按框架要求决定 依赖兼容、构建插件、运行参数都要重新检查

这张表其实就是我的升级优先级判断：先明确收益目标，再决定升级策略。不要为了“统一版本”把 GC、虚拟线程、依赖升级一次性全打开。

3. 实际收益

分代 ZGC

我们对一个缓存型服务升级了 JDK21，并使用分代 ZGC。在没有明显观察到资源占用提高的前提下，解决了 GC 带来的长尾问题。

原始记录里这里有两张内部监控图，一张是发布前后的长尾响应耗时，一张是 JDK11 + G1 与 JDK21 + ZGC 的对比。公开版不放内部截图，只保留结论：

  • 接口响应时间本身不慢，但尾延迟容易被 GC pause 拉高。
  • JDK21 + 分代 ZGC 后，长尾响应耗时有明显收敛。
  • 观察时不能只看 pause time，也要看 CPU、RSS、吞吐。

另外可以参考 Netflix 这篇文章：

https://netflixtechblog.com/bending-pause-times-to-your-will-with-generational-zgc-256629c9386b

虚拟线程

我们在一个 IO 密集型服务里做过全面 JDK21 升级，并进行了虚拟线程灰度验证。

当时观察到的一个现象是：对比非虚拟线程，平台线程数从 2000 左右降低到 300+。中间有一段时间因为某个依赖组件还不支持虚拟线程，内部仍然开了几百个平台线程；后续替换为虚拟线程池后，线程数量继续下降。

接口 P95 的灰度对比大概是：

场景 非虚拟线程 JDK21 P95 虚拟线程 JDK21 P95 降幅
列表查询 284 239 19%
详情查询 589 493 16.3%

加上升级 JDK21 本身带来的优化，对比 JDK11 的应用，虚拟线程 + JDK21 带来的 P95 降幅大概在 20%。

这里需要强调：这个结果只说明“这类 IO 密集型场景值得测”，不代表所有服务开虚拟线程都会变快。

4. 升级前准备

IDE

IDE 要能正确支持 JDK21 的语法、编译、调试和运行配置。版本对应关系直接看官方说明：

https://www.jetbrains.com/help/idea/supported-java-versions.html

原始笔记里有一些本地安装和激活相关内容，公开博客不保留。

Maven

JDK 升级时 Maven 也要检查，尤其是：

  • maven-compiler-plugin 是否支持 JDK21。
  • maven-surefire-plugin / maven-failsafe-plugin 是否能在 JDK21 下正常跑测试。
  • IDE 内置 Maven 与项目 Maven 版本是否一致。
  • 依赖 import 是否正常。

有些问题不是 JDK21 本身导致的，而是 IDE、Maven、插件版本组合导致的。

运行参数

升级时要重点检查旧参数：

  • 旧 GC 参数是否还存在，例如 CMS 相关参数。
  • GC 日志是否迁移到 -Xlog。
  • --add-opens 是否仍然需要。
  • agent 是否兼容 JDK21。
  • 监控、诊断、热修复工具是否依赖 JDK internals。

5. 我的升级建议

我更倾向于拆成几个阶段：

  1. 只升级运行时，不换 GC，不开虚拟线程。
  2. 压测 JDK21 + 原 GC，确认运行时升级本身没有问题。
  3. 单独验证 JDK21 + 分代 ZGC。
  4. 单独验证 JDK21 + 虚拟线程。
  5. 如果要上虚拟线程，必须有独立开关和灰度策略。

这样做的好处是问题好定位。否则 JDK 版本、GC、虚拟线程、依赖升级混在一起，一旦指标异常，很难判断到底是哪一个变量导致的。

如果虚拟线程有问题，优先关闭虚拟线程，不要直接回滚 JDK。

如果 ZGC 表现不符合预期，优先切回原 GC。

只有运行时兼容本身有问题，才需要回退 JDK。

6. 小结

JDK21 升级的价值主要在三件事：

  1. ZGC：解决一类 GC pause 导致的长尾问题。
  2. 虚拟线程：降低 IO 密集型服务的平台线程成本。
  3. 衍生生态：让后续框架、工具链、运行时能力升级更顺。

我的结论比较简单：JDK21 可以作为新的运行时基线去推进，但虚拟线程不要因为“正式发布了”就直接大规模用。JDK 升级和虚拟线程上线是两件事。
