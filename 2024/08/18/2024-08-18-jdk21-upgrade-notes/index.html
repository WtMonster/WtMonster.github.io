JEP 446: Scoped Values (Preview) 21 能在许多场景下替代ThreadLocal，但注意！ 它并不是为了取代 ThreadLocal ！也取代不了
ThreadLocal变量修改难以追踪，难以调试;
容易忘记remove导致内存泄漏;
使用InheritableThreadLocal在子线程中传递父线程变量时， API 要求在其他线程中看不到更改线程局部变量的线程副本，因此会做拷贝逻辑 。

Scoped Value管理的变量不可变，代码更易读，并且通过不可变特性避免了拷贝的内存开销；
作用域更明确，生命周期结束后自然回收，内存更安全。

由于二者设计逻辑不同，依然存在一些场景适配工作量较大，例如将threadLocal当作资源池的场景。
最典型的，例如jackson的资源池，log4j2的ReusableMessage https://github.com/apache/logging-log4j2/pull/1401

这里简单说下，一般升级时需要关注export和opens

export是模块内部会向外export一些包，而不被模块主动export的则认为是随时可变的，不做任何保证。如果使用了模块没有向外export的API，则会在编译期报错，需要在编译和启动参数手动增加export参数。

opens是运行期的检查，对应反射访问模块API的情况，根据运行期间的报错加一下就好，这个需要加在启动参数上。

值得注意的是，在jdk9开始做模块化以后，通过 –illegal-access来做了运行时的模块化控制，也就是opens的统一控制

  • --illegal-access=permit : JDK9以后默认模式，只在第一次非法反射访问时 warning一下

  • --illegal-access=warn ：与 permit类似，不过每次非法反射都会 warning

  • --illegal-access=debug ：在warn的基础上，加上堆栈输出

  • --illegal-access=deny : 拒绝所有非法反射访问内部api，JDK16下的默认模式

但是在JDK17中，通过 JEP 403: Strongly Encapsulate JDK Internals 强封装 ， --illegal-access这个参数被无效化了，相当于只能deny，我们只能通过 opens来控制运行时反射的访问。

JEP 247: Compile for Older Platform Versions

编译时可以通过source和target参数灵活控制产物jdk版本，对兼容老版本还是有好处的。

JEP 238: Multi-Release JAR Files

可以指定不同jdk版本的环境中使用不同的api。例如我维护了一个jar包，其中一个接口在jdk8下希望使用某个api，在jdk21下使用另一个api，就可以通过 Multi-Release JAR 实现，不过这个东西实际比较鸡肋。
