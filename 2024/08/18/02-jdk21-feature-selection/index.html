JDK8 到 JDK21 重要特性梳理

Posted on 2024-08-18 Edited on 2025-07-20 In Java

注意：预览功能非必要不建议使用。

以下表格去除了大部分日常开发不需要感知的特性，按照个人理解根据重要性、影响程度，以及日常开发需要关注的程度进行排序。会比较主观，欢迎补充和指正。

参考入口：

  • https://openjdk.org/jeps/0
  • https://openjdk.org/projects/jdk/17/jeps-since-jdk-11
  • https://openjdk.org/projects/jdk/21/jeps-since-jdk-17
  • https://docs.oracle.com/en/java/javase/21/migrate/significant-changes-jdk-release.html
  • https://github.com/nipafx/demo-java-x

1. 语言

Reference JDK Version Description Project & Tags
JEP 395: Records 16 充当不可变数据的载体的类，与元组形式类似，拥有极简单的类结构。有一些序列化相关特性需要注意。 Amber - language
JEP 430: String Templates (Preview) 21 提供字符串模板和嵌入表达式生成字符串的方式。这个方向很有意思，但后来打算重做，这也是我不建议开启预览功能的原因之一。 Amber - language、performance
JEP 361: Switch Expressions / JEP 394 / JEP 440 / JEP 441 14 / 16 / 21 / 21 对 switch 表达式、instanceof 类型转换、Record 模式匹配都做了简化。嵌套 Record 的场景会比较明显。 Amber - language
JEP 286: Local-Variable Type Inference 10 var 可以用，但我建议只在特定情境下使用：不关心具体类型，或者可以在上下文中明确直接看到类型信息。比如 Stream 里某些很长的泛型类型。 Amber - language
JEP 378: Text Blocks 15 可以使用三个双引号包含多行字符串。写 SQL、JSON、测试数据会舒服很多。 Amber - language
JEP 409: Sealed Classes 17 密封类，可以限制哪些其他类或接口可以继承或实现它们。 Amber - specification
JEP 431: Sequenced Collections 21 提供了一套有序集合接口标准。不是大特性，但补上了集合框架里长期不太顺手的一块。 Amber - specification
JEP 309: Dynamic Class-File Constants 11 提供类似 invokedynamic 的常量创建形式，一种使用场景是通过这种方式实现高性能懒加载单例。 Valhalla
JEP 443: Unnamed Patterns and Variables (Preview) 21 对于没有用的变量可以用 _ 代替命名表示省略。JDK21 中仍然是 Preview。 Amber - language

语言特性我个人最容易用起来的是 Records、Text Blocks、switch/pattern matching、Sequenced Collections。Preview 功能可以看，但不建议作为生产主链路基础。

2. JVM & Compiler

Reference JDK Version Description
ZGC(JEP377) & Shenandoah(JEP379) 15 ZGC 和 Shenandoah 发布了生产版本。
JEP 439: Generational ZGC 21 ZGC 支持分代了，提升很大，但是需要充分压测。
JEP 328: Flight Recorder 11 绝对的神器，已经 backport 到 jdk8u262，现在不少 JDK8 服务也可以用。JDK 升级、GC 选型、虚拟线程灰度都应该用 JFR 看证据。
JEP 307: Parallel Full GC for G1 10 我原始笔记里写过“G1 支持并发 fullgc”，更准确地说是 G1 Full GC 并行化。16GB 堆在一些场景下也可以压到秒级以内。
JEP 158: Unified JVM Logging / JEP 271: Unified GC Logging 9 规范化了虚拟机日志和 GC 日志，升级时旧 GC 日志参数要迁移到 -Xlog。
JEP 345: NUMA-Aware Memory Allocation for G1 14 支持对 NUMA 架构的感知，特定场景下有意义。
JEP 280: Indify String Concatenation 9 编译时通过 invokedynamic 实现字符串拼接，以前是 StringBuilder，也为多语言铺路。
JEP 312: Thread-Local Handshakes 10 线程局部握手，某些 VM 操作不用总等全局安全点。
JEP 451: Prepare to Disallow the Dynamic Loading of Agents 21 JDK 准备在将来的版本中默认禁止对正在运行的 JVM 动态加载 agent。监控、诊断、热修复相关工具都要提前验证。
JEP 346: Promptly Return Unused Committed Memory from G1 12 G1 会更好地返还内存给操作系统，但不一定对所有业务有感知。
JEP 344: Abortable Mixed Collections for G1 12 G1 能够通过划分必须回收区域、非必需回收区域来更精确地控制 MixedGC 耗时。
JEP 318: Epsilon: A No-Op Garbage Collector 11 无 GC 收集器，这玩意一般用不上，更多是实验和测试场景。
JEP 416: Reimplement Core Reflection with Method Handles 18 使用 MethodHandles 重新实现反射。部分场景变快，部分场景变慢，主要目的是规范化。顺带一提，Java 明显想尽量减少 C 代码，替换成 Java 实现。

这部分才是 JDK 升级时更应该关注的地方。语言特性只是表面，运行时行为变化才决定上线风险。

3. Other

Reference JDK Version Description Project & Tags
JEP 444: Virtual Threads 21 唯一真神，虚拟线程。但 JDK21 上坑比较多，暂不适用于可靠性要求较高的生产环境直接大规模使用。 Loom
JEP 446: Scoped Values (Preview) 21 能在许多场景下替代 ThreadLocal，但注意，它并不是为了完全取代 ThreadLocal，也取代不了所有用法。 Loom
JEP 453: Structured Concurrency (Preview) 21 结构化并发提供任务组的结构化组织方式，和虚拟线程适配很好。JDK21 里还是 Preview。 Loom
JEP 412: Foreign Function & Memory API 17 Panama 对 JDK 的影响其实比 Loom 可能更大，只是对上层开发者的影响不及 Loom。JDK21 阶段仍在预览演进，后续 JDK22 才稳定。 Panama
JEP 448: Vector API (Sixth Incubator) 21 支持 SIMD 向量化加速计算。对 JSON 解析、压缩、数值计算等基础库更有价值。 Panama
JEP 358: Helpful NullPointerExceptions 14 能够更准确地提供空指针异常信息。 -
JEP 321: HTTP Client API 11 JDK 提供了一套标准 HTTP Client API。 -
JEP 349: JFR Event Streaming 14 JFR 事件流，可以实现持续线上监控。我对虚拟线程 pin 问题的监控就是基于这个方向做的。 -
JEP 259: Stack-Walking API 9 在 StackWalker 之前只能通过 getStackTrace 获取完整线程堆栈，StackWalker 允许只处理需要的 StackFrame。诊断工具和框架比较有用。 -
JEP 250 / JEP 310 / JEP 341 / JEP 350 9 / 10 / 12 / 13 CDS/AppCDS，主要问题是需要提前构建好存档，有点麻烦。 -
JEP 193: Variable Handles 9 通过 VarHandle 可以实现更细粒度的内存屏障控制，例如一些高性能缓存库会用。 Code Tools
JEP 254: Compact Strings 9 String 底层从 char 变成 byte，内存占用可能减少，但特定场景下可能增加 CPU 开销。 -
JEP 285: Spin-Wait Hints 9 自旋场景可以使用 Thread.onSpinWait() 给 CPU 提示。但注意它不是性能银弹，而且命中 C2 前基本等于啥也没干。 -

4. 废弃和删除

Reference JDK Version Description
JEP 374: Deprecate and Disable Biased Locking 15 偏向锁被废弃并默认关闭。大部分应用不用管，但老参数要清理。
JEP 363: Remove the Concurrent Mark Sweep (CMS) Garbage Collector 14 CMS 移除了。还在用 CMS 参数的服务会直接启动失败。
JEP 366: Deprecate the ParallelScavenge + SerialOld GC Combination 14 一些旧 GC 组合不再推荐。
JEP 410: Remove the Experimental AOT and JIT Compiler 17 移除了实验性质的 AOT 和 Graal JIT。需要 AOT/JIT 相关能力的直接看 GraalVM 或后续 Leyden。
JEP 214: Remove GC Combinations Deprecated in JDK 8 9 一些 JDK8 已废弃的 GC 组合被删除。

5. JDK8 到 JDK21 升级时我会重点看什么

  1. GC 参数，尤其是 CMS、GC 日志、-Xlog。
  2. 强封装，尤其是反射访问 JDK internals 的地方。
  3. agent 行为，尤其是动态 attach。
  4. JFR 是否能作为常态化观测手段。
  5. 虚拟线程是否真的适合当前服务，不要只看宣传。

结论还是那句话：JDK21 的价值不只是语法，真正影响生产升级的是运行时能力、观测能力和兼容性风险。
