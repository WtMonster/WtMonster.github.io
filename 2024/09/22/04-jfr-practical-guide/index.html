JFR 实战记录

Posted on 2024-09-22 Edited on 2025-10-18 In Java

本文最早是基于 JDK11 整理的，对于 JDK14 的 JFR Event Streaming，以及 JDK16 引入的 jdk.ObjectAllocationSample 事件等不做过多展开。

JFR 是我在做 JDK 升级、GC 选型、虚拟线程灰度时最常用的工具之一。它不是只有性能专家才需要用，业务开发也应该会基本的采集和分析。

什么是 JFR

JFR 是 Java Flight Recorder，前身是 JRockit Flight Recorder，是 JVM 内置的、基于事件的 JDK 监控记录框架。

这个命名参考了黑匣子对于飞机的作用，把 Java 进程比喻成飞机飞行。顾名思义，这个记录主要用于问题定位和持续监控。

官方描述：

https://openjdk.org/jeps/328

JFR 基于事件思想的起源：

https://openjdk.org/jeps/167

JFR 事件大全：

https://bestsolution-at.github.io/jfr-doc/openjdk-11.html

JFR 是一个目标用于线上持续监控的特性。在配置正确的情况下，开销可以很低。JDK11 以前它是商业特性，2018 年 Java Flight Recorder 开源并作为 OpenJDK 11 的一部分发布。

JDK8u262 后 JFR 被反向移植：

  • https://mail.openjdk.org/pipermail/jdk8u-dev/2020-July/012143.html
  • https://bugs.openjdk.org/browse/JDK-8223147

这意味着在 JDK8u262 以后的版本也可以自由使用 JFR。当然，在某些供应商的 OpenJDK8 中可能更早被免费开放，比如 Azul。

JFR 具有以下关键特性：

  • 低开销。配置正确时，可以在生产环境核心业务进程中长期运行，也可以随时开启与关闭。
  • 可以查看出问题时间段内的数据，而不是只能看某个瞬时状态。
  • 可以同时分析 Java 应用、JVM 内部以及当前 Java 进程运行环境。
  • 基于事件采集，可以分析比较底层的信息，例如对象分配、方法采样、热点方法、调用堆栈、安全点、锁占用、GC、JIT、CodeCache。
  • 有完善 API，用户可以自定义事件生产与消费。

为什么要使用 JFR

存在的问题

  1. 目前通过 JMX 对 JVM 的监控粒度比较粗。线上出现非业务逻辑问题后，一般只能通过 dump 堆快照结合日志分析。对 JVM 的详细把控只能通过 dump 获得瞬时状态，而且 dump 到的数据不一定准确。

  2. 从业务层面看，真实业务逻辑通常比较复杂，部分老代码可读性低，常常难以直观感知性能瓶颈。

  3. 从组件框架层面看，框架和组件封装层次比较深，日常开发又忙于需求，接触层次主要集中在业务代码层面，导致很难定位到框架或组件代码中可能存在的性能瓶颈。

  4. 业务研发和组件研发的感知经常是割裂的。组件开发人员感知集中在组件代码层面，容易忽略业务场景下的调用特征；业务开发人员感知集中在业务层面，又很难直观看到组件层面的瓶颈。

  5. 对应用的优化通常以代码重构和架构重构为主，但重构代价大、周期长，有时候通过消除代码层面的瓶颈来提升应用性能，也是一个不错的选择。

使用 JFR 的好处

  1. 定位问题直达本质。JFR 的思想是万物皆为 event。这些 event 可以是 JVM 事件，例如进入 safepoint、STW、锁争用、内存分配、线程上下文切换，也可以是自定义事件。它们直接反映了应用在 JVM 和操作系统层面的根本开销。

  2. JFR 监控开销很低。相比一些外部 profiler，JFR 更适合线上长期采集。一些常用监控手段，比如 jstack 和 jmap，需要让 JVM 进入 safepoint 状态才能获取线程堆栈和内存使用情况，会对线上应用有影响。JFR 在 JVM 内部进行数据采集，可以做到比较低的性能开销。

  3. 使用 JFR 有助于形成对 JVM 更全面的认知。日常开发中，JVM 往往是一个只暴露少数指标的黑盒。JFR 能让我们看到更具体的事件和时间线。

JFR 的使用场景

  1. 分析应用瓶颈。本地分析可以通过压测模拟生产流量，或者回放生产流量。这种方式下可以采用比较激进的 jfc 配置，对应用进行更全面的事件采集。

  2. 线上性能监控、辅助问题定位。线上可以使用 default.jfc 或自定义配置进行低开销事件采集，通过 dump 一段时间内的 JFR 记录来分析应用瓶颈和问题。

JFR 历史

JFR 0.9 版本对应 JDK7 和 JDK8。JDK7u40 之后，实现了和 JRockit Flight Recorder 一样的功能，并添加了各项数据配置，用来打开或关闭一些统计数据功能。JDK8u40 之后，可以在运行时灵活打开关闭 JFR。

JFR 1.0 版本对应 JDK9 和 JDK10。在这一版本之后，增加了 JFR 事件接口，用户可以生产或消费某种事件。

JFR 2.0 版本对应 JDK11，这也是我原始文档主要基于的版本。

JDK14 推出了 JFR Event Streaming，让用户处理 JFR 事件更加灵活方便。

常见参数：

key 默认值 含义
delay 0 延迟多久后启动 JFR 记录，支持 60s、20m、1h、1d 这类单位。一般为了避免框架初始化影响，可以延迟 1 分钟开始记录。
disk true 是否写入磁盘。
dumponexit false 程序退出时，是否 dump 出 .jfr 文件。
duration 0 JFR 记录持续时间，0 代表不限制，一直记录。
filename - dump 输出文件。
name 无 记录名称。可以启动多个 JFR 记录，这个名称用于区分。
maxage 0 只在 disk=true 时有效。最大文件记录保存时间。
maxsize 250MB 左右 只在 disk=true 时有效。最大文件大小。
path-to-gc-roots false 是否记录 GC Roots 到活动对象的路径。一般不打开，怀疑内存泄漏时再短期开启。
settings default.jfc 事件采集配置。JDK 自带 default.jfc 和 profile.jfc。也可以指定自己的 jfc 文件。

path-to-gc-roots 不建议默认开启。首先这个在我个人定位问题时很少用到，其次它成本比较高。一般通过对象分配堆栈就能定位，如果定位不到，怀疑有其他引用，例如 ThreadLocal 没有释放，再在 dump 时采集 GC Roots。

-XX:FlightRecorderOptions

参数 默认值 含义
allow_threadbuffers_to_disk false 是否允许在线程 buffer 阻塞时直接将 thread buffer 内容写入文件。一般没必要开启。
globalbuffersize 根据 memorysize 自动计算 单个 global buffer 大小，一般通过 memorysize 设置，不建议自己设置。
maxchunksize 12M 存入磁盘的每个临时文件大小。不能小于 1M，也不要比 memorysize 或 globalbuffersize 小。
memorysize 10M JFR global buffer 占用的整体内存大小。
numglobalbuffers 根据 memorysize 自动计算 global buffer 个数，一般通过 memorysize 设置，不建议自己设置。
old-object-queue-size 256 Old Object Sample 事件记录多少个 Old Object。
repository java.io.tmpdir JFR 保存到磁盘的临时记录位置。
retransform true 是否通过 JVMTI 转换 JFR 相关 Event 类，一般不用改。
samplethreads true 是否开启线程采集状态位。
stackdepth 64 事件堆栈采集深度。不要设置过大，事件种类多时会影响性能。
threadbuffersize 8KB thread buffer 大小，最好不要随意修改。

JFR 自定义配置文件

建议关闭 OldObjectSample 事件采集，它可能导致内存问题。如果需要开启，建议在压测或镜像环境接入流量观察几天，看 RSS 是否正常。

JDK 中自带两份 jfc 配置：

  • default.jfc：低开销配置，可以安全地在生产环境中连续使用，官方目标通常小于 1%。
  • profile.jfc：适合分析瓶颈，官方声称开销低于 2%。

对于 profile.jfc，经过实测，性能开销不止官方宣称的 2%，不适合用于生产环境长期开启。

JFR 中有一些指标监控已经在 JMX 中存在，对于一些异常和 ERROR 的记录也能通过现有 log 分析，因此不必重复记录。

另外一些事件记录开销比较大，例如 ObjectAllocationInNewTLAB 和 ObjectAllocationOutsideTLAB。也有一些事件对应用问题排查参考意义不大，需要调整或关闭，在需要采集时再动态开启。

此外，有一部分事件采集频率需要根据应用情况适当调整。

Jeyzer 的 jfc 配置可以参考：

JFR 文件拆分

我们可能一次性 dump 了比较多的记录，或者想要一次性 dump 一段比较长时间的记录进行分析。但是 JMC 对 JFR 记录的分析十分消耗 CPU 和内存，所以对于较大的 JFR 记录需要拆分。

原始记录里这里有一组内部压测截图，公开版不放内部图，只保留测试结论。

测试条件是在使用 JVM 参数开启 JFR 记录、应用已经平稳运行的情况下，测试 dump 文件、停止 JFR、开启 JFR 对应用的影响。

主要通过查看执行 JFR 指令时 CPU load 和 response time 的抖动情况来评估。

测试应用是一个缓存型服务，接口响应时间快、性能高，能承受比较大的并发。如果 JFR 会导致比较大的开销，比较容易在响应时间上看出来。

当时设置了远高于线上流量的单机请求量，一方面让 JFR 记录到更多事件，放大 JFR 开销；另一方面测试系统压力较大时 JFR 对系统的影响。

结论：

  1. dump 5 分钟数据，没有观察到系统明显抖动。
  2. stop JFR，没有观察到明显抖动。
  3. start JFR，也没有观察到系统明显抖动或开启后性能降低。

从原理角度也比较好解释：JFR 是持续记录的，dump 文件时只是拉取一段时间内已有记录，不需要在 dump 时再进行太多分析。
