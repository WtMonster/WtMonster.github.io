Loom 虚拟线程相关问题记录

Posted on 2024-09-05 Edited on 2025-09-10 In Java

JDK21 的虚拟线程目前存在许多严重问题，尽量不要在可靠性要求较高的服务中直接大规模使用。

这篇基本是我当时整理 Loom 问题时的笔记。不是虚拟线程入门，也不是“开箱即用教程”，重点是风险、死锁、监控和适配。

简单介绍

虚拟线程使用一个专用的、FIFO 模式下的 ForkJoin 线程池。parallelStream 使用的是公共 ForkJoinPool，模式不一样。

它的并行度就是承载虚拟线程的平台线程数，默认与可用核心数相等，也可以使用参数调整：

虚拟线程的核心价值是：阻塞时可以从平台线程上卸载，让平台线程继续执行其他虚拟线程。所以它适合 IO 密集型场景，而不是 CPU 密集型场景。

风险

虚拟线程支持 ThreadLocal 和 synchronized，但是存在一些缺陷，有一些问题需要额外注意。

1. 绑定问题

  1. 虚拟线程执行时，在 synchronized 块或方法中执行代码，会与平台线程绑定。同步块内，以及等待同步锁都会 pin。因此 JDK21 里需要尽量使用 ReentrantLock 替代。这个问题在 JDK24 的 JEP 491 里已经从底层解决了一大块。

  2. 虚拟线程执行 native 代码或外来函数时是无法被中止的。这个暂时无法完全从底层解决，典型场景之一是类初始化。

2. 死锁：这块风险在 JEP 中实际上被低估了

最经典的场景，大部分死锁本质都是这类问题：synchronized 内外出现同一资源竞争。

大体原理是：

Lock 内部实现的 AQS 使用 FIFO 队列来进行线程排队和资源抢占。对于已经在队列中的线程来说，无论是公平锁还是非公平锁，都遵循 FIFO。

死锁场景是：FIFO 队列中，一个阻塞在同步块外部 lock 上的虚拟线程排在前面，一个阻塞在同步块内部 lock 上的虚拟线程排在后面。当某个线程释放 lock 时，会把队列前面的虚拟线程 unpark，然后自身马上停靠在 synchronized 同步块上被绑定。

此时其他虚拟线程都在 pinned 状态，占用了所有平台线程。这个 unpark 是无效的，因为被唤醒的虚拟线程没有平台线程可以绑定，也就无法继续抢锁；而真正能推进状态的线程又排在队列后面，最终卡住。

ThreadLocal

ThreadLocal 的问题主要在于线程模型变化。

没有虚拟线程前，我们可以假定线程数不会太多。但是虚拟线程打破了这个限制，一个系统可能同时存在非常多虚拟线程。

因此 ThreadLocal 的数据量就可能非常非常多，这也就是 ThreadLocal 的风险所在。用的时候注意一下就好。

这里 cue 一下 ScopedValue。有些人可能期望使用 ScopedValue 完全替代 ThreadLocal，其实这是不现实的。

最典型的是 ThreadLocal 用于资源池化的场景，例如 Jackson 和 Log4j2 中都有这种用法。Log4j2 的 ReusableMessage 后来也老实换成普通资源池了：

https://github.com/apache/logging-log4j2/pull/1401

监控

1. JFR

可以使用 JFR 来观察虚拟线程在绑定时的挂起情况。

几个相关事件：

  • jdk.VirtualThreadStart：记录虚拟线程启动，默认关闭。
  • jdk.VirtualThreadEnd：记录虚拟线程结束，默认关闭。
  • jdk.VirtualThreadPinned：虚拟线程绑定在平台线程上时，如果阻塞超过阈值，会产生这个事件，默认开启。
  • jdk.VirtualThreadSubmitFailed：启动或唤醒虚拟线程失败，默认开启。

官方说明：

https://openjdk.org/jeps/425#JDK-Flight-Recorder-JFR

这里要注意，这个参数在 JDK21 上存在过一些严重问题。例如：

  • https://bugs.openjdk.org/browse/JDK-8322846
  • https://bugs.openjdk.org/browse/JDK-8325521

所以它适合上线前排查，不适合生产长期打开。

解决措施

最好的解决方案是官方提供 synchronized 对虚拟线程的支持。JDK24 中已经实现了适配：

https://openjdk.org/jeps/491

如果当前运行时还是 JDK21，可以考虑：

  1. 尽量从 synchronized 迁移到 ReentrantLock。
  2. 不要在 synchronized 内做 IO、sleep、park、等待连接池。
  3. 对于 ConcurrentHashMap 这类替换成本较高的场景，不要机械替换。
  4. 由于 JFR event 并不包含持有 Monitor 的帧，所以上线前排查阶段使用 -Djdk.tracePinnedThreads 会比较方便，但只建议非生产使用。

排查时重点关注：

  1. sync 块内外出现同一资源竞争，这种可能造成死锁。
  2. sync 里面有 park，但是没有内外资源竞争，这种主要影响性能。
  3. sync 里面没有 park 操作，无竞争时影响不大，有竞争时会导致性能变差，可以通过 JFR 事件 jdk.JavaMonitorEnter 观察。
  4. 其他绑定问题，例如 native、FFI、类加载、agent。

Tips

虚拟线程不再需要池化。虚拟线程非常轻量，随用随建就行。

但是线程池部分思想依然有效，例如使用线程池的思想控制访问资源的并发量。

这种场景下，内部可以用 Semaphore 简单控制并发量，而不需要复杂的线程管理逻辑。

也就是说，虚拟线程池的目的不再包含资源复用，而应该只包含一些更轻量的附加控制。
