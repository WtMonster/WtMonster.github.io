从 JDK21 看到 JDK25：Leyden、JFR Method Timing 与虚拟线程后续演进

Posted on 2026-05-05 Edited on 2026-05-03 In Java

这篇是 JDK21 升级之后继续关注的一些方向，主要包括虚拟线程后续演进、JFR、GC、对象头压缩，以及 Leyden。

JDK21 是一个 LTS 落点，但很多能力并不是到 JDK21 就结束了。实际做升级时，还是要看后续几个版本在解决什么问题。

1. JDK21 之后比较重要的变化

Loom

Release JEP 我的理解
24 JEP 491: Synchronize Virtual Threads without Pinning 解决虚拟线程在 synchronized 上 pinning 的主要问题。JDK21 中虚拟线程最大的一类坑，后续版本已经在底层处理。
25 JEP 506: Scoped Values Scoped Values 正式交付。它提供基于显式作用域的值传递方式，适合替代一部分 ThreadLocal 上下文传递场景。

JDK21 里虚拟线程已经正式可用，但如果从生产大规模使用的角度看，我认为 JDK24 的 JEP 491 非常关键。

JVM 相关

Release JEP 我的理解
25 JEP 519: Compact Object Headers Lilliput 项目的阶段性成果，把对象头压缩到更小。对象很多的应用可能会有内存收益，具体效果要看应用对象分布。
25 JEP 521: Generational Shenandoah Shenandoah 也进入分代方向。低延迟 GC 后面基本都在利用分代假说继续优化。
25 JEP 520: JFR Method Timing & Tracing JFR 支持更细的方法级耗时 tracing。这个对线上分析很有吸引力，但生产开启范围和开销要认真控制。
23 / 24 JEP 474: ZGC: Generational Mode by Default / JEP 490: ZGC: Remove the Non-Generational Mode ZGC 后续只有分代模式。JDK21 先引入分代 ZGC，后面继续默认化、移除非分代模式。

这几个点说明一个趋势：JVM 还在持续降低延迟、降低内存占用、增强观测能力。JDK21 不是终点，只是一个比较适合落地的版本。

2. Leyden 相关进展

已交付

  • JEP 483: Ahead-of-Time Class Loading & Linking
  • JEP 514: Ahead-of-Time Command-Line Ergonomics
  • JEP 515: Ahead-of-Time Method Profiling
  • JEP 516: Ahead-of-Time Object Caching with Any GC

这里和我最初整理时有一个变化：JEP 516 已经是 JDK26 delivered，不再是 doing 状态。所以如果只讨论 JDK25，需要把它放到“后续继续关注”的位置；如果讨论 Leyden 路线，就可以一起看。

继续关注

  • AOT Code Compilation 相关草案。
  • 对更多 class loader 场景的支持。
  • 和 Spring Boot 打包、jar 展开、CDS 的工程化结合。
  • 启动参数、训练流程和正式启动流程的隔离。

3. 核心 JEP 解读

JEP 483：Ahead-of-Time Class Loading & Linking

JEP 483 的核心是存储类加载和链接的训练数据。它关注的是应用启动时反复做的类加载、解析、链接工作，能不能提前做一部分。

需要注意的是，它不是把 Java 变成 native 可执行文件，也不是直接消除 JVM 的动态性。它更像是让 JVM 在启动时少做一部分重复工作。

我的理解：

  • 对启动速度和预热速度有帮助。
  • 对 lambda、框架启动、类加载较多的应用可能更明显。
  • class loader 越复杂，落地越需要验证。

JEP 515：Ahead-of-Time Method Profiling

JEP 515 存储方法调用 profile，例如方法执行次数、调用路径等，让 JIT 在应用启动时就能知道一些历史执行情况。

这个点很有意思。理想情况下，热点路径在首次触达时就可以更快进入较好的编译优化状态。

但这里也有一个需要注意的地方：如果 JIT 更早知道热点路径，启动初期可能会更积极地做优化，CPU 资源消耗也可能更集中。所以即便有 AOT Method Profiling，流量接入也不一定能完全取消慢启动，只是慢启动可以更快、更可控。

未来如果 AOT Code Compilation 继续推进，上面这个问题可能会继续改善。

JEP 514：Ahead-of-Time Command-Line Ergonomics

JEP 514 主要是改善命令行体验，把训练/生成配置的流程包装得更顺一点。

这个 JEP 本身不算最核心的性能能力，但对工程化很重要。AOT 相关能力如果使用成本太高，就很难真正进入普通服务发布流程。

一个实际需要注意的问题是：训练阶段和正式启动阶段不要简单复用所有启动参数。比如管理端口、监控端口、临时目录、文件路径，都可能在训练生成阶段产生冲突。工程化时最好把“训练配置”和“正式运行配置”拆开。

JEP 516：Ahead-of-Time Object Caching with Any GC

JEP 516 是 JDK26 的内容，不属于 JDK25，但它很能说明 Leyden 的方向：不只是类加载和链接，后续还会把一些对象状态提前缓存。

这个方向如果成熟，对启动和预热会有更大帮助。但具体能缓存什么、和应用框架如何结合、如何保证正确性，都需要继续观察。

4. 目前比较大的问题

class loader

Leyden 相关能力很容易受 class loader 影响。

简单 classpath 应用是一回事，真实服务经常还有：

  • fat jar / nested jar。
  • 应用服务器 class loader。
  • 插件化 class loader。
  • 框架自定义 class loader。
  • agent 或字节码增强。

所以不要只看简单 demo 的效果。真实服务要看自己的打包方式和类加载路径。

Spring Boot 这类框架已经在 jar 展开、CDS、AOT 方向做了一些工作，普通服务优先看框架集成会更稳。

相关资料：

  • https://spring.io/blog/2024/08/29/spring-boot-cds-support-and-project-leyden-anticipation
  • https://docs.spring.io/spring-boot/reference/packaging/efficient.html
  • https://docs.spring.io/spring-boot/docs/3.2.9/reference/html/executable-jar.html#appendix.executable-jar.nested-jars

GC 支持

早期 Leyden/AOT 能力和 GC 的配合有一些限制，后续 JEP 516 已经推进到 Any GC。这个方向需要继续跟。

对业务应用来说，比较理想的状态是：不需要为了启动优化放弃原本选择的 GC。

训练流量

AOT profile 类能力依赖训练数据。训练数据是否代表真实流量，会直接影响收益。

如果训练流量和真实流量差异很大，可能只优化了启动路径，没优化真正的业务热点路径。对复杂服务来说，训练阶段如何设计是一个工程问题，不只是 JVM 参数问题。

5. Leyden 和 Native Image 的区别

Leyden 相比 Native Image，不会直接砍掉 JVM 的动态性。对普通 Java 应用来说，这一点很重要。

Native Image 更像是基于 closed-world 假设，把应用提前编译成原生可执行文件。它启动快、内存占用低，但对反射、动态代理、运行时代码生成、类加载等能力有更多约束。

Leyden 的方向更像是：保留 JVM 运行时，同时把一部分可以提前做的事情提前做。比如类加载链接、方法 profile、对象缓存，未来可能还有代码编译。

所以我理解它们不是简单替代关系：

  • Native Image 更激进，收益明显，约束也明显。
  • Leyden 更温和，更适合大量依赖 JVM 动态性的传统 Java 服务。

相关背景可以看：

  • https://openjdk.org/projects/leyden/notes/01-beginnings
  • https://mail.openjdk.org/pipermail/discuss/2020-April/005429.html
  • https://mail.openjdk.org/pipermail/leyden-dev/2022-May/000005.html

6. 我后面会重点关注什么

  1. JEP 491 后，虚拟线程在真实服务里的风险是否明显降低。
  2. JFR Method Timing & Tracing 的生产使用方式和开销。
  3. Compact Object Headers 对对象密集型服务的实际内存收益。
  4. ZGC/Shenandoah 分代化之后的真实吞吐和延迟表现。
  5. Leyden 和 Spring Boot、CDS、jar 展开的集成路径。
  6. AOT profile/缓存能力对启动、预热、扩容速度的改善。

7. 小结

JDK21 升级完以后，不代表这条线结束。

JDK21 是一个适合落地的版本；JDK24/JDK25/JDK26 展示的是后续演进方向：虚拟线程更成熟、JFR 更强、GC 更偏分代化、对象布局更紧凑、Leyden 开始改善启动和预热。

对应用团队来说，不需要盲目追每一个版本，但需要知道这些变化在解决什么问题。这样下一次升级就不是临时补课，而是持续演进。
