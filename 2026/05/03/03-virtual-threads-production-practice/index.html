虚拟线程生产实践：收益、pinning 风险与排查方法

Posted on 2026-05-03 In Java

JDK21 的虚拟线程目前仍然有一些需要认真对待的风险。我的建议是：可以评估、可以灰度、可以在合适的 IO 密集型服务里用，但不要在可靠性要求很高的核心服务里一上来就大规模使用。

这篇不是虚拟线程入门，而是我在做 JDK21 升级时对虚拟线程风险、监控和适配方式的整理。

1. 简单介绍

虚拟线程使用一个专用的 ForkJoin 线程池，调度模式和 parallelStream 使用的公共 ForkJoinPool 不一样。

它的并行度就是承载虚拟线程的平台线程数，默认与可用核心数相关，也可以通过参数调整：

虚拟线程的核心价值是：阻塞时可以从平台线程上卸载，让平台线程继续执行其他虚拟线程。所以它适合 IO 密集型场景，而不是 CPU 密集型场景。

虚拟线程也不再需要池化。虚拟线程很轻量，随用随建就行。

但是线程池的一部分思想依然有效，例如控制访问资源的并发量。这类场景不应该再用复杂线程池复用线程，而是用 Semaphore、限流器、连接池等方式控制资源并发。

3. 风险

虚拟线程支持 ThreadLocal 和 synchronized，但支持不代表没有代价。JDK21 里主要需要注意两类问题：

  1. pinning，也就是虚拟线程被绑定在平台线程上。
  2. ThreadLocal 在线程数量模型变化后的放大问题。

3.1 绑定问题

JDK21 中，虚拟线程在 synchronized 块或 synchronized 方法中执行阻塞操作时，会和平台线程绑定。同步块内阻塞、等待同步锁，都可能 pin。

另外，虚拟线程执行 native 代码或 foreign function 时也无法正常卸载。这类问题更难从业务代码表面看出来，典型场景之一是类加载。

JEP 491 在 JDK24 解决了 synchronized 导致虚拟线程 pinning 的主要问题，但如果你的生产运行时是 JDK21，就仍然要按 JDK21 的行为处理。

4. 死锁风险

这块风险在很多介绍里容易被低估。大部分死锁本质上都可以归到一类问题：同步块内外存在同一资源竞争，同时同步块可能 pin 住平台线程。

  • 同步块外部会竞争某个资源；
  • 同步块内部也会竞争同一个资源；
  • 同步块内的阻塞会 pin 住承载它的平台线程；
  • 平台线程数量是有限的。

当所有平台线程都被 pinned 虚拟线程占住时，某个已经被 unpark 的虚拟线程可能没有平台线程可以继续执行。队列里真正能推进状态的虚拟线程又排在后面，最终就可能卡住。

如果简化成一句话：sync 块内外出现同一资源竞争，是虚拟线程下非常需要小心的结构。

4.1 单线程调度器下更容易理解

更简单的理解方式是：如果只有一个 carrier thread，vt1 拿到了 lock A 后遇到 IO 让出，vt2 进入 synchronized 后又去拿 lock A，并且在同步块里 pin 住唯一 carrier thread，那 vt1 就没机会回来释放 lock A。

现实里 carrier thread 不止一个，所以不是每次都复现，但结构是一样的。

4.2 Native 或类加载导致的问题

native/FFI 导致的 pinning 触发概率可能没那么高，但原理类似。比如类加载过程中和类加载外存在同一个锁、同一个资源竞争，就可能出现很难排查的问题。

所以虚拟线程适配时，不能只扫业务里的 synchronized，还要关注依赖库、agent、驱动、类加载路径。

5. ThreadLocal

ThreadLocal 的问题主要在于线程模型变了。

没有虚拟线程前，我们通常可以假定线程数不会太多。但虚拟线程打破了这个限制，一个系统可能同时存在非常多虚拟线程。

因此 ThreadLocal 的数据量就可能非常大，这就是 ThreadLocal 的风险所在。用的时候注意一下就好，但不能完全忽略。

这里也提一下 Scoped Values。有些人可能期望用 Scoped Values 完全替代 ThreadLocal，其实不现实。Scoped Values 更适合不可变上下文传递，但 ThreadLocal 被当作资源池、对象缓存、临时 buffer 使用时，就不是简单替换的问题。

比较典型的风险场景：

  • ThreadLocal 存大对象。
  • ThreadLocal 没有 remove。
  • 使用 InheritableThreadLocal 传递上下文。
  • 日志、JSON、序列化、链路追踪框架里隐式使用 ThreadLocal。
  • 把 ThreadLocal 当成资源池。

虚拟线程上线前，这些地方都应该扫一遍。

6. 监控

6.1 JFR

可以使用 JFR 来观察虚拟线程在绑定时的挂起情况。

几个事件需要重点看：

  • jdk.VirtualThreadStart：记录虚拟线程启动，默认关闭。
  • jdk.VirtualThreadEnd：记录虚拟线程结束，默认关闭。
  • jdk.VirtualThreadPinned：虚拟线程 pinned 在平台线程上时，如果阻塞超过阈值，会产生这个事件，默认开启。
  • jdk.VirtualThreadSubmitFailed：启动或唤醒虚拟线程失败，默认开启。
  • jdk.JavaMonitorEnter：monitor 竞争事件，辅助观察 synchronized。

官方说明可以参考：

  • https://openjdk.org/jeps/425#JDK-Flight-Recorder-JFR
  • https://openjdk.org/jeps/349

6.3 jdk.tracePinnedThreads

-Djdk.tracePinnedThreads 只能作为非生产排查手段，不建议长期在线上开。

full 会打印完整堆栈，并突出 native frame 和持有 monitor 的 frame。
short 只输出关键帧。

这个参数很适合上线前排查 synchronized，但不要把它当成生产监控方案。生产监控还是优先用 JFR。

7. 解决和适配思路

最好的解决方案当然是官方从底层解决 synchronized 与虚拟线程 pinning 的问题。JDK24 的 JEP 491 已经往这个方向走了。

如果当前运行时还是 JDK21，可以考虑：

  1. 尽量从 synchronized 迁移到 ReentrantLock，尤其是同步块里会阻塞的场景。
  2. 不要在 synchronized 内做 IO、sleep、park、等待连接池。
  3. 对于 ConcurrentHashMap 这类替换成本较高的场景，不要机械替换，需要看实际数据结构和性能影响。
  4. 上线前用 JFR 和 jdk.tracePinnedThreads 找出明显 pinning 点。
  5. 依赖库版本要单独验证，不要假设所有库都天然适配虚拟线程。

排查时重点关注：

  • sync 块内外出现同一资源竞争，这种可能造成死锁。
  • sync 里面有 park，但没有内外资源竞争，这种主要影响性能。
  • sync 里面没有 park，无竞争时影响不大，有竞争时可能导致性能变差。
  • native/FFI/类加载/agent 相关绑定问题。

8. 线程转储

9. 小结

虚拟线程很有价值，但 JDK21 上的生产落地不能只看收益。

我的建议：

  • IO 密集型服务可以优先试点。
  • 可靠性要求高的核心服务要谨慎。
  • synchronized、ThreadLocal、依赖库、agent 都要排查。
  • 生产监控优先用 JFR。
  • jdk.tracePinnedThreads 只做非生产排查。

如果只能记住一句话：虚拟线程不是“开了就一定变快”的开关，它改变了线程成本模型，也会把过去隐藏在线程、锁和 ThreadLocal 里的问题放大。
