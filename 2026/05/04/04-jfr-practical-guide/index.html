JFR 实战：低开销观测 JVM 与定位性能问题

Posted on 2026-05-04 Edited on 2026-05-03 In Java

本文主要整理 JFR 的基本概念、使用方式和一些生产使用时需要注意的点。它不是完整的 JFR API 文档，更像是一份实践手册。

1. 什么是 JFR

JFR 是 Java Flight Recorder，是 JVM 内置的、基于事件的 JDK 监控记录框架。这个名字可以理解成“Java 进程的黑匣子”，主要用于问题定位和持续监控。

官方描述见 JEP 328。

JFR 基于事件思想，GC、锁竞争、对象分配、线程状态、安全点、JIT 编译、异常、自定义事件等，都可以被记录为事件。

JFR 的几个关键特性：

  • 低开销。配置正确时，可以用于生产环境长期运行。
  • 可以分析某个问题时间段，而不是只能拿到某个瞬时 dump。
  • 能同时观察 Java 应用、JVM 内部和进程运行环境。
  • 基于事件采集，可以分析比较底层的信息，例如对象分配、热点方法、锁占用、安全点、GC、JIT、CodeCache。
  • 有 API，可以自定义事件生产和消费。

JFR 在 JDK11 开源并进入 OpenJDK。JDK8u262 以后也有 JFR backport，所以不少 JDK8 环境也能使用。

JDK14 引入 JFR Event Streaming，让用户可以更灵活地消费 JFR 事件。这个能力在虚拟线程 pinning 监控里很有用。

2. 为什么要使用 JFR

2.1 现有监控的问题

很多系统平时主要依赖外部指标、日志、JMX、线程 dump、堆 dump 来排查问题。

这些工具都有价值，但也有明显不足：

  1. JMX 粒度比较粗，很多 JVM 内部行为看不到。
  2. 线程 dump、堆 dump 更像是瞬时快照，容易错过问题发生时的真实状态。
  3. dump 操作可能需要让 JVM 进入 safepoint，对线上应用有影响。
  4. 业务日志通常只能看到业务层面的问题，很难定位到 JVM 层面的根本开销。
  5. 框架封装层次深时，靠读代码不一定能快速感知真正的性能瓶颈。

2.2 JFR 的好处

JFR 的好处在于它能把很多 JVM 层面的事件放到一条时间线上。

例如：

  • 进入 safepoint。
  • STW。
  • 锁争用。
  • 内存分配。
  • 线程上下文切换。
  • GC。
  • JIT 编译。
  • 异常。
  • 自定义事件。

这些事件直接反映了应用在 JVM 和操作系统层面的开销。通过 JFR 聚合、火焰图和时间轴，可以从 JVM 层面反向定位到代码层面。

在做 JDK 升级时，这一点尤其重要。因为升级以后很多变化不是业务代码引起的，而是运行时、GC、JIT、线程模型、依赖库行为变化引起的。

3. JFR 历史简单整理

  • JDK7/JDK8 时代：JFR 还带有商业特性背景。
  • JDK11：JFR 开源并成为 OpenJDK 的一部分。
  • JDK14：推出 JFR Event Streaming。
  • JDK8u262：JFR 被 backport 到 OpenJDK 8u。

所以现在看 JFR，不应该再把它当成一个很重的 profiler，而应该把它当作 JVM 自带的低开销事件系统。

4. 工具

4.1 JMC

JMC 是分析 JFR 的常用图形化工具，源码在这里：

  • https://github.com/openjdk/jmc

本地分析时，我更推荐使用 JMC 打开 .jfr 文件。IDE 自带的 JFR 视图有时功能不完整，或者不同版本表现不一致。

打开 JFR 后，我一般会先看：

  • Method Profiling：热点方法。
  • Memory：对象分配和堆使用。
  • Lock Instances：锁竞争。
  • GC：GC 周期和暂停。
  • Threads：线程状态。
  • Exceptions：异常事件。

不要只看最终汇总。JFR 最有价值的是时间轴，最好结合问题发生的时间窗口看。

4.2 jcmd

jcmd 是线上动态控制 JFR 最常用的命令。它可以启动、dump、停止、检查 JFR 记录。

4.3 jfr

jfr 命令用于处理 .jfr 文件，例如拆分、打印、查看 summary。官方文档见：

  • https://docs.oracle.com/en/java/javase/21/docs/specs/man/jfr.html

5. 方式一：使用 JVM 参数启动

例如：

常用参数：

key 默认值 含义
delay 0 延迟多久后启动 JFR 记录。支持 60s、20m、1h、1d。为了避开应用启动噪声，可以延迟一段时间。
disk true 是否写入磁盘。长期记录一般建议打开。
dumponexit false 程序退出时是否 dump 出 .jfr 文件。
duration 0 记录持续时间。0 表示不限制持续时间。
filename 空 输出文件路径。
name 无 记录名称。可以启动多个 JFR 记录，所以最好设置名称。
maxage 0 最长保存多久的数据。只在 disk=true 时有效。
maxsize 250MB 左右 最大文件大小。只在 disk=true 时有效。
path-to-gc-roots false 是否记录 GC Roots 路径。一般不打开，怀疑内存泄漏时再短期开启。
settings default.jfc 事件采集配置。JDK 自带 default.jfc 和 profile.jfc。

path-to-gc-roots 不建议默认开启，因为它成本比较高，通常只有怀疑内存泄漏时才需要。

6. 其他配置：FlightRecorderOptions

常见的 -XX:FlightRecorderOptions 配置：

key 默认值 含义
memorysize 10M JFR global buffer 总内存大小。一般通过这个参数控制即可。
globalbuffersize 自动计算 单个 global buffer 大小，不建议手动设置。
numglobalbuffers 自动计算 global buffer 个数，不建议手动设置。
maxchunksize 12M 写入磁盘的每个临时 chunk 大小。不要比 buffer 小太多。
repository 临时目录 JFR 临时文件保存位置。
stackdepth 64 事件堆栈深度。不要盲目调大，会增加开销。
threadbuffersize 8KB thread buffer 大小。一般不建议修改。
samplethreads true 是否开启线程采样相关能力。

实际使用时，不建议一开始就调很多底层参数。先用默认值，出现文件太大、事件丢失、buffer 不够时再调整。

8. 方式三：JMC 直接连接

本地分析时，JMC 直接连接 JVM 也很方便。它适合开发、压测、问题复现环境。

生产环境我更倾向于长期低开销记录，然后按问题窗口 dump 文件出来分析。这样对生产进程的操作更可控。

9. JFR 自定义配置文件

JDK 自带两份配置：

  • default.jfc：低开销配置，适合生产环境长期使用。
  • profile.jfc：事件更多，适合分析瓶颈，但不建议长期在线上开启。

我的建议：

  1. 生产长期记录用保守配置。
  2. profile.jfc 更适合短时间压测或问题复现。
  3. ObjectAllocationInNewTLAB、ObjectAllocationOutsideTLAB 这类事件开销不低，不要默认开得太激进。
  4. Old Object Sample 谨慎开启。怀疑内存泄漏时再短时间开启。
  5. 已经能从外部监控或日志稳定拿到的信息，不一定要在 JFR 里重复高频采集。
  6. 根据应用特征调整阈值，而不是所有应用用一份配置。

JFR 的低开销不是无条件的。配置越激进、事件越多、堆栈越深，开销和文件体积都会上升。

10. JFR 文件拆分

有时一次 dump 的记录比较大，JMC 打开会比较消耗 CPU 和内存。这时可以用 JDK 自带的 jfr 命令拆分。

拆分后逐个打开会轻松很多。

11. 开启 JFR 对性能的影响

我做过的验证结论是：在使用保守配置、持续记录已经平稳运行的情况下，dump 一段时间窗口的数据通常不会造成明显抖动。

原因也比较好理解：JFR 是持续记录的，dump 时主要是把已有记录拉出来，不是在 dump 时才重新分析所有数据。

但这不代表可以随便开。验证 JFR 影响时建议看：

  • CPU load。
  • 响应时间。
  • 文件大小。
  • JFR 事件量。
  • dump、start、stop 操作发生前后的指标变化。

如果应用非常敏感，还是要先在压测环境验证。

12. 和虚拟线程结合

虚拟线程灰度时，JFR 是首选观测手段。

重点事件：

  • jdk.VirtualThreadPinned
  • jdk.VirtualThreadStart
  • jdk.VirtualThreadEnd
  • jdk.VirtualThreadSubmitFailed
  • jdk.JavaMonitorEnter

命令行 dump 文件适合事后分析，Event Streaming 更适合持续监控。JDK14 之后可以通过 RecordingStream 消费事件，把 pinning 事件接入日志或指标系统。

13. 小结

JFR 不是只有性能专家才用的工具。只要涉及 JDK 升级、GC 切换、虚拟线程灰度、线上性能问题，JFR 都值得放进标准流程里。

我的使用习惯是：

  1. 平时保守配置长期记录。
  2. 问题发生后按时间窗口 dump。
  3. JMC 看时间轴，不只看汇总。
  4. 压测和灰度阶段用 JFR 做升级前后对比。
  5. 虚拟线程场景用 JFR 事件持续观察 pinning。
