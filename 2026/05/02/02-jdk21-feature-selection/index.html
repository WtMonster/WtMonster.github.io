JDK21 特性取舍：哪些值得生产关注，哪些暂缓

Posted on 2026-05-02 Edited on 2026-05-03 In Java

注意：预览功能非必要不建议使用。

这篇是对 JDK8 到 JDK21 之间一些特性的整理。下面的表格去除了大部分日常开发不需要感知的特性，并且按照我个人理解，根据重要性、影响程度、日常开发需要关注的程度做了排序，会比较主观。

完整列表还是建议看官方：

  • JEP Index
  • JEPs in JDK 17 integrated since JDK 11
  • JEPs in JDK 21 integrated since JDK 17
  • Oracle JDK 21 migration guide

1. 语言和日常开发相关

Reference JDK Version 我的理解
JEP 395: Records 16 充当不可变数据的载体，与元组形式类似，类结构很简单。DTO、事件对象、配置快照都比较适合。有一些序列化相关特性需要注意。
JEP 361: Switch Expressions / JEP 394 / JEP 440 / JEP 441 14 / 16 / 21 / 21 对 switch 表达式、instanceof 类型转换、Record 模式匹配都做了简化。这个对代码可读性提升比较明显，但别把复杂业务全塞到一个 switch 里。
JEP 286: Local-Variable Type Inference 10 var 可以用，但我建议只在上下文中能直接看出类型，或者具体类型不重要时使用。比如 Stream 里某些很长的泛型类型。
JEP 378: Text Blocks 15 多行字符串，写 SQL、JSON、测试数据会舒服很多。动态 SQL 还是别靠字符串拼接。
JEP 409: Sealed Classes 17 密封类，可以限制哪些类或接口能继承/实现它。和 pattern matching 配合起来会比较自然。
JEP 431: Sequenced Collections 21 提供了一套“有序集合”的标准接口。这个不是大特性，但补上了集合框架里长期不太顺手的一块。
JEP 430: String Templates (Preview) 21 字符串模板这个方向很有意思，但在 JDK21 里是 Preview，后来也有重做方向。我不建议生产主链路开启。
JEP 443: Unnamed Patterns and Variables (Preview) 21 未使用变量可以用 _ 表示。语义上是好事，但 JDK21 中仍是 Preview。

上面这些里面，我认为 Records、Text Blocks、switch/pattern matching、Sequenced Collections 是可以逐步进入日常编码习惯的。Preview 功能可以学习，不建议作为生产代码的基础。

2. JVM 和 Compiler 相关

Reference JDK Version 我的理解
ZGC / Shenandoah 15 ZGC 和 Shenandoah 发布生产版本。真正对 JDK21 升级有吸引力的是后面的分代 ZGC。
JEP 439: Generational ZGC 21 ZGC 支持分代了，提升很大。适合 GC 延迟敏感应用重点验证，但仍然要看吞吐、CPU、RSS。
JEP 328: Flight Recorder 11 绝对的神器。JDK 升级、GC 选型、虚拟线程灰度都应该用 JFR 看证据。后面会单独写 JFR 实战。
JEP 307: Parallel Full GC for G1 10 G1 的 Full GC 支持并行后，极端情况下的暂停问题会比以前好一些。
JEP 158 / JEP 271 9 统一 JVM 日志和统一 GC 日志。升级时旧 GC 日志参数要迁移到 -Xlog。
JEP 345: NUMA-Aware Memory Allocation for G1 14 NUMA 场景下有意义，普通开发不一定直接感知。
JEP 280: Indify String Concatenation 9 字符串拼接底层改成 invokedynamic。业务一般不用管，但说明高版本 JDK 在底层持续优化。
JEP 312: Thread-Local Handshakes 10 某些 VM 操作不再总是需要全局安全点，对停顿控制有帮助。
JEP 451: Prepare to Disallow the Dynamic Loading of Agents 21 未来会限制运行时动态加载 agent。监控、诊断、热修复相关工具都要提前验证。
JEP 416: Reimplement Core Reflection with Method Handles 18 用 MethodHandles 重新实现反射。主要目的是规范化，部分场景变快，部分场景变慢，需要靠压测和 JFR 看。

这部分才是升级 JDK 时更应该关注的地方。语言特性只是表面，运行时行为变化才决定上线风险。

3. Loom、Panama 和其他能力

Reference JDK Version 我的理解
JEP 444: Virtual Threads 21 虚拟线程是 JDK21 最有吸引力的特性之一。IO 密集型服务值得重点看，但 JDK21 的 pinning 风险不能忽略。
JEP 446: Scoped Values (Preview) 21 能在一些场景替代 ThreadLocal，但注意，它不是为了完全取代 ThreadLocal，也取代不了所有用法。
JEP 453: Structured Concurrency (Preview) 21 提供任务组的结构化组织方式，和虚拟线程适配很好。JDK21 里还是 Preview，生产主链路谨慎。
JEP 412: Foreign Function & Memory API 17 Panama 对 JDK 的影响可能比 Loom 更底层，只是对上层业务开发没有那么直接。
JEP 448: Vector API (Incubator) 21 SIMD 向量化。对 JSON 解析、压缩、数值计算这类基础库更有价值，普通业务不一定直接用。
JEP 349: JFR Event Streaming 14 JFR 事件流，可以实现持续线上监控。虚拟线程 pinning 监控就很适合用它做。
JEP 259: Stack-Walking API 9 比 getStackTrace 更灵活，可以按需处理 StackFrame。诊断工具和框架比较有用。
JEP 250 / JEP 310 / JEP 341 / JEP 350 9 / 10 / 12 / 13 CDS/AppCDS，可以改善启动和类加载成本，但要提前构建归档，工程化成本要考虑。
JEP 193: Variable Handles 9 更细粒度的内存屏障控制，很多高性能库会用。
JEP 254: Compact Strings 9 String 底层从 char 变成 byte，内存占用可能减少，但特定场景可能增加 CPU 开销。
JEP 285: Spin-Wait Hints 9 自旋场景可以用 Thread.onSpinWait() 给 CPU 提示，不过要理解它不是业务性能银弹。

这里面我最关注的是 Virtual Threads、JFR Event Streaming 和 CDS。Panama/Vector API 很重要，但更偏底层库和基础设施。

4. 废弃和删除

升级时经常不是新特性出问题，而是老东西不见了。

Reference JDK Version 影响
JEP 374: Deprecate and Disable Biased Locking 15 偏向锁被废弃并默认关闭。大部分应用不用管，但老参数要清理。
JEP 363: Remove the Concurrent Mark Sweep (CMS) Garbage Collector 14 CMS 移除了。还在用 CMS 参数的服务会直接启动失败。
JEP 366: Deprecate the ParallelScavenge + SerialOld GC Combination 14 一些旧 GC 组合不再推荐。
JEP 410: Remove the Experimental AOT and JIT Compiler 17 移除了实验性质的 AOT 和 Graal JIT。需要 AOT/JIT 相关能力的直接看 GraalVM 或后续 Leyden。
JEP 214: Remove GC Combinations Deprecated in JDK 8 9 一些 JDK8 已废弃的 GC 组合被删除。

这部分建议在升级前直接扫启动参数。很多服务的 JVM 参数是历史遗留，没人确定它现在是否还有意义。

5. 模块化和兼容

JDK9 之后模块化是升级时必须关注的内容。

简单说：

  • exports 影响编译期和链接期访问。
  • opens 影响运行期反射访问。
  • 如果用了未导出的 JDK 内部 API，编译期可能报错。
  • 如果反射访问了未开放的包，运行期可能报错。

JDK9 开始有 --illegal-access 做运行时缓冲：

  • --illegal-access=permit
  • --illegal-access=warn
  • --illegal-access=debug
  • --illegal-access=deny

但到了 JDK17，JEP 403 强封装 JDK internals，--illegal-access 基本不能再作为迁移缓冲。后面只能通过更明确的 --add-opens 控制运行时反射访问。

我的建议是：

  1. 能不用 JDK 内部 API 就不用。
  2. 必须用的时候，最小范围加 --add-opens 或 --add-exports。
  3. 不要直接复制一大串参数到所有应用。

6. 适配和兼容相关

JEP 247: Compile for Older Platform Versions 提供 --release，可以更可靠地控制编译目标平台。维护多 JDK 版本兼容库时比较有用。

JEP 238: Multi-Release JAR Files 可以在不同 JDK 版本下使用不同 class。理论上很灵活，但实际使用要谨慎，复杂度不低。

还有 Jakarta 迁移也要关注。很多老项目依赖过去 Java EE 相关包，升级时可能需要显式引入依赖，或者迁移坐标。

7. 我的取舍

如果只从生产升级角度看，我会这么排：

优先级 特性
必看 分代 ZGC、JFR、模块强封装、agent 行为变化
值得采用 Records、Text Blocks、switch/pattern matching、Sequenced Collections
单独灰度 Virtual Threads、JFR Event Streaming、CDS
谨慎试点 Scoped Values、Structured Concurrency、FFM、Vector API
先不进主链路 JDK21 中仍是 Preview/Incubator 的能力

结论还是那句话：JDK21 的价值不只是语法，真正影响生产升级的是运行时能力、观测能力和兼容性风险。
