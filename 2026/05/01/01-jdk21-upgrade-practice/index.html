JDK21 升级实践：从收益判断到落地路径

Posted on 2026-05-01 Edited on 2026-05-03 In Java

当前升级 JDK 的动力实际上主要来自三项：ZGC、虚拟线程、衍生生态。

这篇先不做 JDK21 新特性大全，只讨论“什么应用适合升级”“升级到底能拿到什么收益”“实际落地时先关注哪些风险”。特性细节和虚拟线程/JFR 会放到后面的文章单独展开。

1. 总体分析

ZGC

优点：ZGC 基本消除了 GC 带来的延迟，并且对大对象有更好的处理。

不足：ZGC 吞吐量比不上 G1。即便分代 ZGC 的吞吐已经有了很大提升，但与 G1 依然有一定差距，所以不能只看 pause time。

我的判断是：如果应用的主要问题是 GC 导致的长尾，ZGC 很值得测；如果应用本身更看重极限吞吐，或者 GC 本来就不是瓶颈，那升级收益未必明显。

虚拟线程

优点：适用于 IO 密集型应用。一方面可以提高系统性能，另一方面可以极大消除线程带来的内存压力。

不足：JDK21 里的虚拟线程坑还比较多，尤其是 synchronized pinning、ThreadLocal 使用方式变化、依赖库兼容性等问题。可靠性要求很高的应用，不建议一上来就大规模启用。

这里要强调一点：升级到 JDK21 和启用虚拟线程应该拆开。JDK21 可以先升，虚拟线程应该单独灰度。

衍生生态

许多 Java 生态框架都在逐步提高基线版本，低 JDK 版本的维护会越来越弱。如果想探索基于高版本 JDK 的框架和工具链，升级 JDK 基本是绕不开的。

这类收益不一定马上体现在接口耗时上，但会体现在后续框架升级、依赖治理、工具链维护成本上。

2. 什么应用适合升级 JDK

目标 适用应用 具体版本/策略 风险
ZGC GC 延迟敏感的应用，例如缓存应用；本地有大缓存、频繁生成大对象的应用 如果对吞吐追求不高，可以考虑 JDK17 以上；如果对吞吐有更高要求，优先验证 JDK21 分代 ZGC 分代 ZGC 需要充分压测，不要只看暂停时间
虚拟线程 IO 密集型应用，文件 IO 场景需要单独验证 JDK21 可用，但建议独立开关、独立灰度 synchronized pinning 可能降低吞吐，严重时可能造成死锁
衍生生态 需要使用高版本框架、工具链或运行时能力的应用 根据框架要求决定 依赖兼容、构建插件、运行参数都要重新检查

这张表其实是我对升级优先级的判断：先找收益目标，再决定升级策略。不要为了“统一版本”就把所有能力一起打开。

3. 实际收益

分代 ZGC

一个比较典型的场景是缓存型服务。服务本地保留了较多数据，接口响应时间本身不慢，但尾延迟容易被 GC pause 拉高。

这类服务升级到 JDK21 并使用分代 ZGC 后，比较明显的收益是长尾响应耗时收敛。在没有观察到资源占用异常提高的前提下，GC 带来的尾延迟问题得到了缓解。

这里有几个观察点：

  • 不要只看平均耗时，要看 P95/P99。
  • 不要只看 GC pause，要同时看 CPU、RSS、吞吐。
  • 同口径对比很重要，最好是同一类实例、同一类流量、同一段时间窗口。

分代 ZGC 的价值不是“所有服务都会变快”，而是它能解决一类非常明确的问题：GC pause 导致的长尾。

虚拟线程

另一个试点场景是 IO 密集型接口服务。启用虚拟线程后，平台线程数从数千量级降到了数百量级，部分接口 P95 有十几个百分点的改善。

这个结果和预期是吻合的：虚拟线程降低的是阻塞式并发模型下的平台线程成本。大量时间花在远程调用、数据库访问、缓存访问上的服务，更容易看到收益。

但这里不能只看收益。虚拟线程上线前必须看这些东西：

  • 是否有大量 synchronized 块或方法。
  • synchronized 里是否有 IO、sleep、park 或锁竞争。
  • ThreadLocal 是否持有大对象或资源对象。
  • HTTP client、数据库驱动、RPC 框架是否验证过虚拟线程兼容性。
  • 是否能通过 JFR 看到 jdk.VirtualThreadPinned。

我的建议是：虚拟线程不要和 JDK21 升级同时作为一个变量上线。先证明应用在 JDK21 下稳定，再开启虚拟线程灰度。

4. 升级前准备

准备工作不复杂，但容易漏。

IDE 和本地环境

IDE 要能正确支持 JDK21 的语法、编译、调试和运行配置。这里不展开具体版本，直接看 IDE 官方对 JDK 版本的支持矩阵即可。

Maven/Gradle

构建工具和插件要重点看：

  • compiler plugin 是否支持 JDK21。
  • surefire/failsafe 是否能在 JDK21 下正常跑测试。
  • 依赖解析是否受 IDE 或 Maven 版本影响。
  • CI 使用的 JDK 和本地是否一致。

如果本地能编译、CI 不能编译，或者 CI 能编译、运行镜像不能启动，后面所有压测都没有意义。

依赖和启动参数

重点检查几类：

  • 字节码增强、agent、APM、诊断工具。
  • 反射比较重的框架。
  • 数据库驱动、HTTP client、RPC client。
  • 老 JVM 参数，尤其是 GC 参数。
  • --add-opens、--add-exports 相关参数。

JDK9 以后模块化逐步收紧，JDK17 以后强封装影响更明显。升级时遇到非法反射访问，不要一股脑放开所有模块，最好按报错最小化处理。

5. 验证路径

我比较推荐把升级拆成几个阶段：

  1. 只升级运行时，不换 GC，不开虚拟线程。
  2. 跑单元测试、集成测试、启动冒烟。
  3. 压测 JDK21 + 原 GC，确认运行时升级本身没有问题。
  4. 单独验证 JDK21 + 分代 ZGC。
  5. 单独验证 JDK21 + 虚拟线程。
  6. 小流量灰度，每次只打开一个变量。

这样做的好处是问题好定位。否则 JDK 版本、GC、虚拟线程、依赖升级混在一起，一旦指标异常，很难判断到底是哪一个变量导致的。

6. 回滚策略

回滚也要拆变量。

如果虚拟线程有问题，优先关闭虚拟线程，不要直接回滚 JDK。
如果 ZGC 表现不符合预期，优先切回原 GC。
只有运行时兼容本身有问题，才需要回退 JDK。

所以开关设计很重要。虚拟线程、GC、JDK 版本最好都有清晰的灰度和回退路径。

7. 小结

JDK21 升级的价值主要在三件事：

  • ZGC：解决一类 GC pause 导致的长尾问题。
  • 虚拟线程：降低 IO 密集型服务的平台线程成本。
  • 生态：让应用站到新的 Java 生态基线上。

但这些收益都不是自动拿到的。比较稳的做法是：先明确收益目标，再拆变量验证，最后用 JFR 和线上指标证明结果。
