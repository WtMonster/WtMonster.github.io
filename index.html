从 JDK21 看到 JDK25：Leyden、JFR Method Timing 与虚拟线程后续演进

Posted on 2026-05-05 Edited on 2026-05-03 In Java

这篇是 JDK21 升级之后继续关注的一些方向，主要包括虚拟线程后续演进、JFR、GC、对象头压缩，以及 Leyden。

JDK21 是一个 LTS 落点，但很多能力并不是到 JDK21 就结束了。实际做升级时，还是要看后续几个版本在解决什么问题。

1. JDK21 之后比较重要的变化

Loom

Release JEP 我的理解
24 JEP 491: Synchronize Virtual Threads without Pinning 解决虚拟线程在 synchronized 上 pinning 的主要问题。JDK21 中虚拟线程最大的一类坑，后续版本已经在底层处理。
25 JEP 506: Scoped Values Scoped Values 正式交付。它提供基于显式作用域的值传递方式，适合替代一部分 ThreadLocal 上下文传递场景。

JDK21 里虚拟线程已经正式可用，但如果从生产大规模使用的角度看，我认为 JDK24 的 JEP 491 非常关键。

JVM 相关

Release JEP 我的理解
25 JEP 519: Compact Object Headers Lilliput 项目的阶段性成果，把对象头压缩到更小。对象很多的应用可能会有内存收益，具体效果要看应用对象分布。
25 JEP 521: Generational Shenandoah Shenandoah 也进入分代方向。低延迟 GC 后面基本都在利用分代假说继续优化。
25 JEP 520: JFR Method Timing & Tracing JFR 支持更细的方法级耗时 tracing。这个对线上分析很有吸引力，但生产开启范围和开销要认真控制。
23 / 24 JEP 474: ZGC: Generational Mode by Default / JEP 490: ZGC: Remove the Non-Generational Mode ZGC 后续只有分代模式。JDK21 先引入分代 ZGC，后面继续默认化、移除非分代模式。

这几个点说明一个趋势：JVM 还在持续降低延迟、降低内存占用、增强观测能力。JDK21 不是终点，只是一个比较适合落地的版本。

2. Leyden 相关进展

已交付

  • JEP 483: Ahead-of-Time Class Loading & Linking
  • JEP 514: Ahead-of-Time Command-Line Ergonomics
  • JEP 515: Ahead-of-Time Method Profiling
  • JEP 516: Ahead-of-Time Object Caching with Any GC

这里和我最初整理时有一个变化：JEP 516 已经是 JDK26 delivered，不再是 doing 状态。所以如果只讨论 JDK25，需要把它放到“后续继续关注”的位置；如果讨论 Leyden 路线，就可以一起看。

继续关注

  • AOT Code Compilation 相关草案。
  • 对更多 class loader 场景的支持。
  • 和 Spring Boot 打包、jar 展开、CDS 的工程化结合。
  • 启动参数、训练流程和正式启动流程的隔离。

3. 核心 JEP 解读

JEP 483：Ahead-of-Time Class Loading & Linking

JEP 483 的核心是存储类加载和链接的训练数据。它关注的是应用启动时反复做的类加载、解析、链接工作，能不能提前做一部分。

需要注意的是，它不是把 Java 变成 native 可执行文件，也不是直接消除 JVM 的动态性。它更像是让 JVM 在启动时少做一部分重复工作。

我的理解：

  • 对启动速度和预热速度有帮助。
  • 对 lambda、框架启动、类加载较多的应用可能更明显。
  • class loader 越复杂，落地越需要验证。

JEP 515：Ahead-of-Time Method Profiling

JEP 515 存储方法调用 profile，例如方法执行次数、调用路径等，让 JIT 在应用启动时就能知道一些历史执行情况。

这个点很有意思。理想情况下，热点路径在首次触达时就可以更快进入较好的编译优化状态。

但这里也有一个需要注意的地方：如果 JIT 更早知道热点路径，启动初期可能会更积极地做优化，CPU 资源消耗也可能更集中。所以即便有 AOT Method Profiling，流量接入也不一定能完全取消慢启动，只是慢启动可以更快、更可控。

未来如果 AOT Code Compilation 继续推进，上面这个问题可能会继续改善。

JEP 514：Ahead-of-Time Command-Line Ergonomics

JEP 514 主要是改善命令行体验，把训练/生成配置的流程包装得更顺一点。

这个 JEP 本身不算最核心的性能能力，但对工程化很重要。AOT 相关能力如果使用成本太高，就很难真正进入普通服务发布流程。

一个实际需要注意的问题是：训练阶段和正式启动阶段不要简单复用所有启动参数。比如管理端口、监控端口、临时目录、文件路径，都可能在训练生成阶段产生冲突。工程化时最好把“训练配置”和“正式运行配置”拆开。

JEP 516：Ahead-of-Time Object Caching with Any GC

JEP 516 是 JDK26 的内容，不属于 JDK25，但它很能说明 Leyden 的方向：不只是类加载和链接，后续还会把一些对象状态提前缓存。

这个方向如果成熟，对启动和预热会有更大帮助。但具体能缓存什么、和应用框架如何结合、如何保证正确性，都需要继续观察。

4. 目前比较大的问题

class loader

Leyden 相关能力很容易受 class loader 影响。

简单 classpath 应用是一回事，真实服务经常还有：

  • fat jar / nested jar。
  • 应用服务器 class loader。
  • 插件化 class loader。
  • 框架自定义 class loader。
  • agent 或字节码增强。

所以不要只看简单 demo 的效果。真实服务要看自己的打包方式和类加载路径。

Spring Boot 这类框架已经在 jar 展开、CDS、AOT 方向做了一些工作，普通服务优先看框架集成会更稳。

相关资料：

  • https://spring.io/blog/2024/08/29/spring-boot-cds-support-and-project-leyden-anticipation
  • https://docs.spring.io/spring-boot/reference/packaging/efficient.html
  • https://docs.spring.io/spring-boot/docs/3.2.9/reference/html/executable-jar.html#appendix.executable-jar.nested-jars

GC 支持

早期 Leyden/AOT 能力和 GC 的配合有一些限制，后续 JEP 516 已经推进到 Any GC。这个方向需要继续跟。

对业务应用来说，比较理想的状态是：不需要为了启动优化放弃原本选择的 GC。

训练流量

AOT profile 类能力依赖训练数据。训练数据是否代表真实流量，会直接影响收益。

如果训练流量和真实流量差异很大，可能只优化了启动路径，没优化真正的业务热点路径。对复杂服务来说，训练阶段如何设计是一个工程问题，不只是 JVM 参数问题。

5. Leyden 和 Native Image 的区别

Leyden 相比 Native Image，不会直接砍掉 JVM 的动态性。对普通 Java 应用来说，这一点很重要。

Native Image 更像是基于 closed-world 假设，把应用提前编译成原生可执行文件。它启动快、内存占用低，但对反射、动态代理、运行时代码生成、类加载等能力有更多约束。

Leyden 的方向更像是：保留 JVM 运行时，同时把一部分可以提前做的事情提前做。比如类加载链接、方法 profile、对象缓存，未来可能还有代码编译。

所以我理解它们不是简单替代关系：

  • Native Image 更激进，收益明显，约束也明显。
  • Leyden 更温和，更适合大量依赖 JVM 动态性的传统 Java 服务。

相关背景可以看：

  • https://openjdk.org/projects/leyden/notes/01-beginnings
  • https://mail.openjdk.org/pipermail/discuss/2020-April/005429.html
  • https://mail.openjdk.org/pipermail/leyden-dev/2022-May/000005.html

6. 我后面会重点关注什么

  1. JEP 491 后，虚拟线程在真实服务里的风险是否明显降低。
  2. JFR Method Timing & Tracing 的生产使用方式和开销。
  3. Compact Object Headers 对对象密集型服务的实际内存收益。
  4. ZGC/Shenandoah 分代化之后的真实吞吐和延迟表现。
  5. Leyden 和 Spring Boot、CDS、jar 展开的集成路径。
  6. AOT profile/缓存能力对启动、预热、扩容速度的改善。

7. 小结

JDK21 升级完以后，不代表这条线结束。

JDK21 是一个适合落地的版本；JDK24/JDK25/JDK26 展示的是后续演进方向：虚拟线程更成熟、JFR 更强、GC 更偏分代化、对象布局更紧凑、Leyden 开始改善启动和预热。

对应用团队来说，不需要盲目追每一个版本，但需要知道这些变化在解决什么问题。这样下一次升级就不是临时补课，而是持续演进。

JFR 实战：低开销观测 JVM 与定位性能问题

Posted on 2026-05-04 Edited on 2026-05-03 In Java

本文主要整理 JFR 的基本概念、使用方式和一些生产使用时需要注意的点。它不是完整的 JFR API 文档，更像是一份实践手册。

1. 什么是 JFR

JFR 是 Java Flight Recorder，是 JVM 内置的、基于事件的 JDK 监控记录框架。这个名字可以理解成“Java 进程的黑匣子”，主要用于问题定位和持续监控。

官方描述见 JEP 328。

JFR 基于事件思想，GC、锁竞争、对象分配、线程状态、安全点、JIT 编译、异常、自定义事件等，都可以被记录为事件。

JFR 的几个关键特性：

  • 低开销。配置正确时，可以用于生产环境长期运行。
  • 可以分析某个问题时间段，而不是只能拿到某个瞬时 dump。
  • 能同时观察 Java 应用、JVM 内部和进程运行环境。
  • 基于事件采集，可以分析比较底层的信息，例如对象分配、热点方法、锁占用、安全点、GC、JIT、CodeCache。
  • 有 API，可以自定义事件生产和消费。

JFR 在 JDK11 开源并进入 OpenJDK。JDK8u262 以后也有 JFR backport，所以不少 JDK8 环境也能使用。

JDK14 引入 JFR Event Streaming，让用户可以更灵活地消费 JFR 事件。这个能力在虚拟线程 pinning 监控里很有用。

2. 为什么要使用 JFR

2.1 现有监控的问题

很多系统平时主要依赖外部指标、日志、JMX、线程 dump、堆 dump 来排查问题。

这些工具都有价值，但也有明显不足：

  1. JMX 粒度比较粗，很多 JVM 内部行为看不到。
  2. 线程 dump、堆 dump 更像是瞬时快照，容易错过问题发生时的真实状态。
  3. dump 操作可能需要让 JVM 进入 safepoint，对线上应用有影响。
  4. 业务日志通常只能看到业务层面的问题，很难定位到 JVM 层面的根本开销。
  5. 框架封装层次深时，靠读代码不一定能快速感知真正的性能瓶颈。

2.2 JFR 的好处

JFR 的好处在于它能把很多 JVM 层面的事件放到一条时间线上。

例如：

  • 进入 safepoint。
  • STW。
  • 锁争用。
  • 内存分配。
  • 线程上下文切换。
  • GC。
  • JIT 编译。
  • 异常。
  • 自定义事件。

这些事件直接反映了应用在 JVM 和操作系统层面的开销。通过 JFR 聚合、火焰图和时间轴，可以从 JVM 层面反向定位到代码层面。

在做 JDK 升级时，这一点尤其重要。因为升级以后很多变化不是业务代码引起的，而是运行时、GC、JIT、线程模型、依赖库行为变化引起的。

3. JFR 历史简单整理

  • JDK7/JDK8 时代：JFR 还带有商业特性背景。
  • JDK11：JFR 开源并成为 OpenJDK 的一部分。
  • JDK14：推出 JFR Event Streaming。
  • JDK8u262：JFR 被 backport 到 OpenJDK 8u。

所以现在看 JFR，不应该再把它当成一个很重的 profiler，而应该把它当作 JVM 自带的低开销事件系统。

4. 工具

4.1 JMC

JMC 是分析 JFR 的常用图形化工具，源码在这里：

  • https://github.com/openjdk/jmc

本地分析时，我更推荐使用 JMC 打开 .jfr 文件。IDE 自带的 JFR 视图有时功能不完整，或者不同版本表现不一致。

打开 JFR 后，我一般会先看：

  • Method Profiling：热点方法。
  • Memory：对象分配和堆使用。
  • Lock Instances：锁竞争。
  • GC：GC 周期和暂停。
  • Threads：线程状态。
  • Exceptions：异常事件。

不要只看最终汇总。JFR 最有价值的是时间轴，最好结合问题发生的时间窗口看。

4.2 jcmd

jcmd 是线上动态控制 JFR 最常用的命令。它可以启动、dump、停止、检查 JFR 记录。

4.3 jfr

jfr 命令用于处理 .jfr 文件，例如拆分、打印、查看 summary。官方文档见：

  • https://docs.oracle.com/en/java/javase/21/docs/specs/man/jfr.html

5. 方式一：使用 JVM 参数启动

例如：

常用参数：

key 默认值 含义
delay 0 延迟多久后启动 JFR 记录。支持 60s、20m、1h、1d。为了避开应用启动噪声，可以延迟一段时间。
disk true 是否写入磁盘。长期记录一般建议打开。
dumponexit false 程序退出时是否 dump 出 .jfr 文件。
duration 0 记录持续时间。0 表示不限制持续时间。
filename 空 输出文件路径。
name 无 记录名称。可以启动多个 JFR 记录，所以最好设置名称。
maxage 0 最长保存多久的数据。只在 disk=true 时有效。
maxsize 250MB 左右 最大文件大小。只在 disk=true 时有效。
path-to-gc-roots false 是否记录 GC Roots 路径。一般不打开，怀疑内存泄漏时再短期开启。
settings default.jfc 事件采集配置。JDK 自带 default.jfc 和 profile.jfc。

path-to-gc-roots 不建议默认开启，因为它成本比较高，通常只有怀疑内存泄漏时才需要。

6. 其他配置：FlightRecorderOptions

常见的 -XX:FlightRecorderOptions 配置：

key 默认值 含义
memorysize 10M JFR global buffer 总内存大小。一般通过这个参数控制即可。
globalbuffersize 自动计算 单个 global buffer 大小，不建议手动设置。
numglobalbuffers 自动计算 global buffer 个数，不建议手动设置。
maxchunksize 12M 写入磁盘的每个临时 chunk 大小。不要比 buffer 小太多。
repository 临时目录 JFR 临时文件保存位置。
stackdepth 64 事件堆栈深度。不要盲目调大，会增加开销。
threadbuffersize 8KB thread buffer 大小。一般不建议修改。
samplethreads true 是否开启线程采样相关能力。

实际使用时，不建议一开始就调很多底层参数。先用默认值，出现文件太大、事件丢失、buffer 不够时再调整。

7.4 查看正在执行的记录

7.5 查看或修改配置

不带参数时是查看当前配置，传入参数时是修改配置。

8. 方式三：JMC 直接连接

本地分析时，JMC 直接连接 JVM 也很方便。它适合开发、压测、问题复现环境。

生产环境我更倾向于长期低开销记录，然后按问题窗口 dump 文件出来分析。这样对生产进程的操作更可控。

9. JFR 自定义配置文件

JDK 自带两份配置：

  • default.jfc：低开销配置，适合生产环境长期使用。
  • profile.jfc：事件更多，适合分析瓶颈，但不建议长期在线上开启。

我的建议：

  1. 生产长期记录用保守配置。
  2. profile.jfc 更适合短时间压测或问题复现。
  3. ObjectAllocationInNewTLAB、ObjectAllocationOutsideTLAB 这类事件开销不低，不要默认开得太激进。
  4. Old Object Sample 谨慎开启。怀疑内存泄漏时再短时间开启。
  5. 已经能从外部监控或日志稳定拿到的信息，不一定要在 JFR 里重复高频采集。
  6. 根据应用特征调整阈值，而不是所有应用用一份配置。

JFR 的低开销不是无条件的。配置越激进、事件越多、堆栈越深，开销和文件体积都会上升。

10. JFR 文件拆分

有时一次 dump 的记录比较大，JMC 打开会比较消耗 CPU 和内存。这时可以用 JDK 自带的 jfr 命令拆分。

拆分后逐个打开会轻松很多。

11. 开启 JFR 对性能的影响

我做过的验证结论是：在使用保守配置、持续记录已经平稳运行的情况下，dump 一段时间窗口的数据通常不会造成明显抖动。

原因也比较好理解：JFR 是持续记录的，dump 时主要是把已有记录拉出来，不是在 dump 时才重新分析所有数据。

但这不代表可以随便开。验证 JFR 影响时建议看：

  • CPU load。
  • 响应时间。
  • 文件大小。
  • JFR 事件量。
  • dump、start、stop 操作发生前后的指标变化。

如果应用非常敏感，还是要先在压测环境验证。

12. 和虚拟线程结合

虚拟线程灰度时，JFR 是首选观测手段。

重点事件：

  • jdk.VirtualThreadPinned
  • jdk.VirtualThreadStart
  • jdk.VirtualThreadEnd
  • jdk.VirtualThreadSubmitFailed
  • jdk.JavaMonitorEnter

命令行 dump 文件适合事后分析，Event Streaming 更适合持续监控。JDK14 之后可以通过 RecordingStream 消费事件，把 pinning 事件接入日志或指标系统。

13. 小结

JFR 不是只有性能专家才用的工具。只要涉及 JDK 升级、GC 切换、虚拟线程灰度、线上性能问题，JFR 都值得放进标准流程里。

我的使用习惯是：

  1. 平时保守配置长期记录。
  2. 问题发生后按时间窗口 dump。
  3. JMC 看时间轴，不只看汇总。
  4. 压测和灰度阶段用 JFR 做升级前后对比。
  5. 虚拟线程场景用 JFR 事件持续观察 pinning。

先有证据，再下结论。

虚拟线程生产实践：收益、pinning 风险与排查方法

Posted on 2026-05-03 In Java

JDK21 的虚拟线程目前仍然有一些需要认真对待的风险。我的建议是：可以评估、可以灰度、可以在合适的 IO 密集型服务里用，但不要在可靠性要求很高的核心服务里一上来就大规模使用。

这篇不是虚拟线程入门，而是我在做 JDK21 升级时对虚拟线程风险、监控和适配方式的整理。

1. 简单介绍

虚拟线程使用一个专用的 ForkJoin 线程池，调度模式和 parallelStream 使用的公共 ForkJoinPool 不一样。

它的并行度就是承载虚拟线程的平台线程数，默认与可用核心数相关，也可以通过参数调整：

虚拟线程的核心价值是：阻塞时可以从平台线程上卸载，让平台线程继续执行其他虚拟线程。所以它适合 IO 密集型场景，而不是 CPU 密集型场景。

虚拟线程也不再需要池化。虚拟线程很轻量，随用随建就行。

但是线程池的一部分思想依然有效，例如控制访问资源的并发量。这类场景不应该再用复杂线程池复用线程，而是用 Semaphore、限流器、连接池等方式控制资源并发。

3. 风险

虚拟线程支持 ThreadLocal 和 synchronized，但支持不代表没有代价。JDK21 里主要需要注意两类问题：

  1. pinning，也就是虚拟线程被绑定在平台线程上。
  2. ThreadLocal 在线程数量模型变化后的放大问题。

3.1 绑定问题

JDK21 中，虚拟线程在 synchronized 块或 synchronized 方法中执行阻塞操作时，会和平台线程绑定。同步块内阻塞、等待同步锁，都可能 pin。

另外，虚拟线程执行 native 代码或 foreign function 时也无法正常卸载。这类问题更难从业务代码表面看出来，典型场景之一是类加载。

JEP 491 在 JDK24 解决了 synchronized 导致虚拟线程 pinning 的主要问题，但如果你的生产运行时是 JDK21，就仍然要按 JDK21 的行为处理。

4. 死锁风险

这块风险在很多介绍里容易被低估。大部分死锁本质上都可以归到一类问题：同步块内外存在同一资源竞争，同时同步块可能 pin 住平台线程。

  • https://mail.openjdk.org/pipermail/loom-dev/2023-July/005993.html
  • https://www.javaspecialists.eu/archive/Issue302-Virtual-Thread-Deadlocks.html

  • 同步块外部会竞争某个资源；
  • 同步块内部也会竞争同一个资源；
  • 同步块内的阻塞会 pin 住承载它的平台线程；
  • 平台线程数量是有限的。

当所有平台线程都被 pinned 虚拟线程占住时，某个已经被 unpark 的虚拟线程可能没有平台线程可以继续执行。队列里真正能推进状态的虚拟线程又排在后面，最终就可能卡住。

如果简化成一句话：sync 块内外出现同一资源竞争，是虚拟线程下非常需要小心的结构。

4.1 单线程调度器下更容易理解

更简单的理解方式是：如果只有一个 carrier thread，vt1 拿到了 lock A 后遇到 IO 让出，vt2 进入 synchronized 后又去拿 lock A，并且在同步块里 pin 住唯一 carrier thread，那 vt1 就没机会回来释放 lock A。

现实里 carrier thread 不止一个，所以不是每次都复现，但结构是一样的。

4.2 Native 或类加载导致的问题

native/FFI 导致的 pinning 触发概率可能没那么高，但原理类似。比如类加载过程中和类加载外存在同一个锁、同一个资源竞争，就可能出现很难排查的问题。

所以虚拟线程适配时，不能只扫业务里的 synchronized，还要关注依赖库、agent、驱动、类加载路径。

5. ThreadLocal

ThreadLocal 的问题主要在于线程模型变了。

没有虚拟线程前，我们通常可以假定线程数不会太多。但虚拟线程打破了这个限制，一个系统可能同时存在非常多虚拟线程。

因此 ThreadLocal 的数据量就可能非常大，这就是 ThreadLocal 的风险所在。用的时候注意一下就好，但不能完全忽略。

这里也提一下 Scoped Values。有些人可能期望用 Scoped Values 完全替代 ThreadLocal，其实不现实。Scoped Values 更适合不可变上下文传递，但 ThreadLocal 被当作资源池、对象缓存、临时 buffer 使用时，就不是简单替换的问题。

比较典型的风险场景：

  • ThreadLocal 存大对象。
  • ThreadLocal 没有 remove。
  • 使用 InheritableThreadLocal 传递上下文。
  • 日志、JSON、序列化、链路追踪框架里隐式使用 ThreadLocal。
  • 把 ThreadLocal 当成资源池。

虚拟线程上线前，这些地方都应该扫一遍。

6. 监控

6.1 JFR

可以使用 JFR 来观察虚拟线程在绑定时的挂起情况。

几个事件需要重点看：

  • jdk.VirtualThreadStart：记录虚拟线程启动，默认关闭。
  • jdk.VirtualThreadEnd：记录虚拟线程结束，默认关闭。
  • jdk.VirtualThreadPinned：虚拟线程 pinned 在平台线程上时，如果阻塞超过阈值，会产生这个事件，默认开启。
  • jdk.VirtualThreadSubmitFailed：启动或唤醒虚拟线程失败，默认开启。
  • jdk.JavaMonitorEnter：monitor 竞争事件，辅助观察 synchronized。

7. 解决和适配思路

最好的解决方案当然是官方从底层解决 synchronized 与虚拟线程 pinning 的问题。JDK24 的 JEP 491 已经往这个方向走了。

如果当前运行时还是 JDK21，可以考虑：

  1. 尽量从 synchronized 迁移到 ReentrantLock，尤其是同步块里会阻塞的场景。
  2. 不要在 synchronized 内做 IO、sleep、park、等待连接池。
  3. 对于 ConcurrentHashMap 这类替换成本较高的场景，不要机械替换，需要看实际数据结构和性能影响。
  4. 上线前用 JFR 和 jdk.tracePinnedThreads 找出明显 pinning 点。
  5. 依赖库版本要单独验证，不要假设所有库都天然适配虚拟线程。

排查时重点关注：

  • sync 块内外出现同一资源竞争，这种可能造成死锁。
  • sync 里面有 park，但没有内外资源竞争，这种主要影响性能。
  • sync 里面没有 park，无竞争时影响不大，有竞争时可能导致性能变差。
  • native/FFI/类加载/agent 相关绑定问题。

8. 线程转储

JDK21 中使用传统 jstack -l <pid> 不会拿到完整虚拟线程信息。

原因也比较合理：虚拟线程数量可能非常多，原本扁平化的线程 dump 很难组织这些关系。引入结构化并发后更是如此。

9. 小结

虚拟线程很有价值，但 JDK21 上的生产落地不能只看收益。

我的建议：

  • IO 密集型服务可以优先试点。
  • 可靠性要求高的核心服务要谨慎。
  • synchronized、ThreadLocal、依赖库、agent 都要排查。
  • 生产监控优先用 JFR。
  • jdk.tracePinnedThreads 只做非生产排查。

如果只能记住一句话：虚拟线程不是“开了就一定变快”的开关，它改变了线程成本模型，也会把过去隐藏在线程、锁和 ThreadLocal 里的问题放大。

5. 模块化和兼容

JDK9 之后模块化是升级时必须关注的内容。

简单说：

  • exports 影响编译期和链接期访问。
  • opens 影响运行期反射访问。
  • 如果用了未导出的 JDK 内部 API，编译期可能报错。
  • 如果反射访问了未开放的包，运行期可能报错。

JDK9 开始有 --illegal-access 做运行时缓冲：

  • --illegal-access=permit
  • --illegal-access=warn
  • --illegal-access=debug
  • --illegal-access=deny

但到了 JDK17，JEP 403 强封装 JDK internals，--illegal-access 基本不能再作为迁移缓冲。后面只能通过更明确的 --add-opens 控制运行时反射访问。

我的建议是：

  1. 能不用 JDK 内部 API 就不用。
  2. 必须用的时候，最小范围加 --add-opens 或 --add-exports。
  3. 不要直接复制一大串参数到所有应用。

JEP 247: Compile for Older Platform Versions 提供 --release，可以更可靠地控制编译目标平台。维护多 JDK 版本兼容库时比较有用。

JDK21 升级实践：从收益判断到落地路径

Posted on 2026-05-01 Edited on 2026-05-03 In Java

当前升级 JDK 的动力实际上主要来自三项：ZGC、虚拟线程、衍生生态。

这篇先不做 JDK21 新特性大全，只讨论“什么应用适合升级”“升级到底能拿到什么收益”“实际落地时先关注哪些风险”。特性细节和虚拟线程/JFR 会放到后面的文章单独展开。

1. 总体分析

ZGC

优点：ZGC 基本消除了 GC 带来的延迟，并且对大对象有更好的处理。

不足：ZGC 吞吐量比不上 G1。即便分代 ZGC 的吞吐已经有了很大提升，但与 G1 依然有一定差距，所以不能只看 pause time。

我的判断是：如果应用的主要问题是 GC 导致的长尾，ZGC 很值得测；如果应用本身更看重极限吞吐，或者 GC 本来就不是瓶颈，那升级收益未必明显。

虚拟线程

优点：适用于 IO 密集型应用。一方面可以提高系统性能，另一方面可以极大消除线程带来的内存压力。

不足：JDK21 里的虚拟线程坑还比较多，尤其是 synchronized pinning、ThreadLocal 使用方式变化、依赖库兼容性等问题。可靠性要求很高的应用，不建议一上来就大规模启用。

这里要强调一点：升级到 JDK21 和启用虚拟线程应该拆开。JDK21 可以先升，虚拟线程应该单独灰度。

衍生生态

许多 Java 生态框架都在逐步提高基线版本，低 JDK 版本的维护会越来越弱。如果想探索基于高版本 JDK 的框架和工具链，升级 JDK 基本是绕不开的。

这类收益不一定马上体现在接口耗时上，但会体现在后续框架升级、依赖治理、工具链维护成本上。

2. 什么应用适合升级 JDK

目标 适用应用 具体版本/策略 风险
ZGC GC 延迟敏感的应用，例如缓存应用；本地有大缓存、频繁生成大对象的应用 如果对吞吐追求不高，可以考虑 JDK17 以上；如果对吞吐有更高要求，优先验证 JDK21 分代 ZGC 分代 ZGC 需要充分压测，不要只看暂停时间
虚拟线程 IO 密集型应用，文件 IO 场景需要单独验证 JDK21 可用，但建议独立开关、独立灰度 synchronized pinning 可能降低吞吐，严重时可能造成死锁
衍生生态 需要使用高版本框架、工具链或运行时能力的应用 根据框架要求决定 依赖兼容、构建插件、运行参数都要重新检查

这张表其实是我对升级优先级的判断：先找收益目标，再决定升级策略。不要为了“统一版本”就把所有能力一起打开。

3. 实际收益

分代 ZGC

一个比较典型的场景是缓存型服务。服务本地保留了较多数据，接口响应时间本身不慢，但尾延迟容易被 GC pause 拉高。

这类服务升级到 JDK21 并使用分代 ZGC 后，比较明显的收益是长尾响应耗时收敛。在没有观察到资源占用异常提高的前提下，GC 带来的尾延迟问题得到了缓解。

这里有几个观察点：

  • 不要只看平均耗时，要看 P95/P99。
  • 不要只看 GC pause，要同时看 CPU、RSS、吞吐。
  • 同口径对比很重要，最好是同一类实例、同一类流量、同一段时间窗口。

分代 ZGC 的价值不是“所有服务都会变快”，而是它能解决一类非常明确的问题：GC pause 导致的长尾。

虚拟线程

另一个试点场景是 IO 密集型接口服务。启用虚拟线程后，平台线程数从数千量级降到了数百量级，部分接口 P95 有十几个百分点的改善。

这个结果和预期是吻合的：虚拟线程降低的是阻塞式并发模型下的平台线程成本。大量时间花在远程调用、数据库访问、缓存访问上的服务，更容易看到收益。

但这里不能只看收益。虚拟线程上线前必须看这些东西：

  • 是否有大量 synchronized 块或方法。
  • synchronized 里是否有 IO、sleep、park 或锁竞争。
  • ThreadLocal 是否持有大对象或资源对象。
  • HTTP client、数据库驱动、RPC 框架是否验证过虚拟线程兼容性。
  • 是否能通过 JFR 看到 jdk.VirtualThreadPinned。

我的建议是：虚拟线程不要和 JDK21 升级同时作为一个变量上线。先证明应用在 JDK21 下稳定，再开启虚拟线程灰度。

4. 升级前准备

准备工作不复杂，但容易漏。

IDE 和本地环境

IDE 要能正确支持 JDK21 的语法、编译、调试和运行配置。这里不展开具体版本，直接看 IDE 官方对 JDK 版本的支持矩阵即可。

Maven/Gradle

构建工具和插件要重点看：

  • compiler plugin 是否支持 JDK21。
  • surefire/failsafe 是否能在 JDK21 下正常跑测试。
  • 依赖解析是否受 IDE 或 Maven 版本影响。
  • CI 使用的 JDK 和本地是否一致。

如果本地能编译、CI 不能编译，或者 CI 能编译、运行镜像不能启动，后面所有压测都没有意义。

依赖和启动参数

重点检查几类：

  • 字节码增强、agent、APM、诊断工具。
  • 反射比较重的框架。
  • 数据库驱动、HTTP client、RPC client。
  • 老 JVM 参数，尤其是 GC 参数。
  • --add-opens、--add-exports 相关参数。

JDK9 以后模块化逐步收紧，JDK17 以后强封装影响更明显。升级时遇到非法反射访问，不要一股脑放开所有模块，最好按报错最小化处理。

5. 验证路径

我比较推荐把升级拆成几个阶段：

  1. 只升级运行时，不换 GC，不开虚拟线程。
  2. 跑单元测试、集成测试、启动冒烟。
  3. 压测 JDK21 + 原 GC，确认运行时升级本身没有问题。
  4. 单独验证 JDK21 + 分代 ZGC。
  5. 单独验证 JDK21 + 虚拟线程。
  6. 小流量灰度，每次只打开一个变量。

这样做的好处是问题好定位。否则 JDK 版本、GC、虚拟线程、依赖升级混在一起，一旦指标异常，很难判断到底是哪一个变量导致的。

6. 回滚策略

回滚也要拆变量。

如果虚拟线程有问题，优先关闭虚拟线程，不要直接回滚 JDK。
如果 ZGC 表现不符合预期，优先切回原 GC。
只有运行时兼容本身有问题，才需要回退 JDK。

所以开关设计很重要。虚拟线程、GC、JDK 版本最好都有清晰的灰度和回退路径。

7. 小结

JDK21 升级的价值主要在三件事：

  • ZGC：解决一类 GC pause 导致的长尾问题。
  • 虚拟线程：降低 IO 密集型服务的平台线程成本。
  • 生态：让应用站到新的 Java 生态基线上。

但这些收益都不是自动拿到的。比较稳的做法是：先明确收益目标，再拆变量验证，最后用 JFR 和线上指标证明结果。
